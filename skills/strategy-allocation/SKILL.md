# テスト戦略: 配分と階層化

テストレベル([`levels.md`](../levels/SKILL.md))で粒度を選んだあと、どのレベルに何本置き、どの軸でテストを階層に割り、何から手厚くするかを決める戦略をまとめる。
ここは戦略の**前半**(どのテストをどれだけ・どう割り・どの順で作るか)を扱う。
緑をどの環境とデータで動かし、どう守り続けるか(シフト・環境・データ・フレーキー・CI ゲート)は後半の [`strategy-operations.md`](../strategy-operations/SKILL.md) にある。

配分・階層化・優先順位は「どのテストを何本どこに、どの順で置くか」の戦略だが、テスト戦略はそれだけでは閉じない。
どこに差し込み(シフト)、どの環境とデータで動かし(環境・データ)、緑をどう守り続けるか(フレーキー対策・CI ゲート)まで含めて初めて戦略になる。
それらは [`strategy-operations.md`](../strategy-operations/SKILL.md) を参照。

## テストピラミッドとテストトロフィー(配分論)

どのレベルに何本書くかの配分を示す 2 つのモデル。
どちらも「遅く脆いテストを上に積みすぎるな」が共通の主張で、重心の置き方が違う。

テストピラミッド(Mike Cohn)は、土台に速い単体テストを多く、上に行くほど結合や E2E を少なく積む。
実行が速く失敗箇所を切り分けやすいが、モックに頼りすぎると「単体は全部緑なのに繋ぐと壊れる」状態を招きうる。

テストトロフィー(Kent C. Dodds)は、結合テストに最も重みを置く。
「ユーザーの使い方に近いほど確信度が高い」という主張で、実依存をなるべく本物にした結合層を厚くする。
ただし結合を厚くするほど 1 本が遅く不安定になりやすく、土台の静的検査(型、lint)で下支えする前提が要る。

実務上の指針はひとつ。
本数ではなく、各テストが「壊れたら本当に困ること」を検証しているかで配分を決める。
速くて安定なものを多く、遅くて脆いものを少なく置く。
これはどちらのモデルとも矛盾しない。

### 網羅の配分

各レベルの網羅対象は [`levels.md`](../levels/SKILL.md) で定義した。
配分はその対象を踏まえ、本数ではなく「壊れたら困ること」で決める。
ロジック分岐の網羅は速く安定な単体に厚く集約し、接続のずれは結合に代表で、要約的な要件確認は上層に薄く置く。
遅く脆いレベル(システム、E2E)へ網羅を持ち上げると、確信度の割に CI が遅く不安定になる。
重要フローほど上層に最小本数だけ置き、その細部の網羅は下層へ送り返す、を全レベルで徹底する。

### 遂行手順(着手→完了)

配分は「振る舞いごとに、どのレベルへ何を置くかを1件ずつ決める」作業だ。台帳([`test-extract-template.md`](../test-extract/assets/test-extract-template.md))の各 `T-ID` を相手に次の順で進める。

1. **各 `T-ID` の検証内容を「ロジック分岐 / 接続のずれ / 要件の通し確認」のどれかに分類する**:同値分割・境界値・条件網羅のような入力空間のロジックか、コンポーネント間や外部依存との繋ぎ目か、ユーザー視点の通しフローか。1つに割り切れないものは複数レベルへ分けて書く(下層で細部、上層で代表)。
2. **分類からレベルを割り当てる**:ロジック分岐 → 単体(速く安定な土台に厚く)、接続のずれ → 結合に代表1本、要件の通し確認 → システム/E2E に最小本数。割り当てたレベルを台帳の手法/レベル欄に書く。
3. **「壊れたら本当に困ること」で本数の濃淡を付ける**:同じレベル内でも、壊れたときの影響が大きい `T-ID`(後述のリスクベースで上位)に手厚く、それ以外は代表1本に絞る。本数の目標値そのものを目的にしない。
4. **上層に持ち上がった網羅を下層へ送り返す**:E2E や システムに入力の同値分割・境界の網羅が紛れていたら、その細部を単体へ移し、上層には代表フロー1本だけ残す。遅く脆い層に網羅を溜めない。
5. **配分の形をモデルと突き合わせて点検する**:単体が薄く E2E が厚い逆ピラミッドになっていないか、または結合を厚くしたなら土台の静的検査(型・lint)で下支えできているか(トロフィー前提)を確認する。重心が「速く安定なものを多く、遅く脆いものを少なく」に向いていれば、どちらのモデルでも妥当。

### 完了チェック(もれ確認)

- **全 `T-ID` にレベルが割り当たったか**:台帳の各行にレベル欄が埋まっているか。空欄は配分未決(`grep -E '^\| T-' <台帳>` でレベル欄の空きを目視)。
- **逆ピラミッドになっていないか**:単体の本数 < E2E の本数 になっていないか。なっていれば網羅が上層に溜まっている証拠で、手順4の送り返しが未了。
- **上層に細部網羅が残っていないか**:システム/E2E の `T-ID` に、入力の同値分割・境界値が混ざっていないか。混ざっていれば下層へ移す。
- **濃淡が影響と整合するか**:手厚く本数を割いた `T-ID` が、リスク上位(高影響)のものと一致しているか。低影響に厚く高影響に薄いなら配分が逆。

## 関心事による階層化(粒度とは別の軸)

テストレベルは**粒度**(対象の広さ: 単体→結合→E2E)で段を分ける軸だった。
それと直交するもう一本の軸が、**関心事**でテストを階層に割る考え方である。
同じ E2E の中にも「URL が正しく描画されるか」「入力検証が受入仕様を満たすか」「変数割り当てが正しいか」という別々の関心が同居しており、これらを一枚のテストに混ぜると、何か一つ変わるたびに無関係な検証まで巻き込んで赤くなる。

### 分割の軸
関心事で割るとき、次の3つを手がかりにする。

- **完結できる単位で切る**:それ単体で合否を言い切れる最小のまとまり(1リクエストの検証処理、1関数の入出力、1コールツリー)を1階層にする。隣の関心を借りないと判定できないなら、まだ混ざっている。
- **境界付けられたコンテキストで切る**:ドメインの文脈境界に沿って分ける。セッションをまたぐ一連操作、単一 URL のレスポンス、変数割り当て、はそれぞれ別コンテキストで、別階層が受け持つ。
- **フローと状態で切る**:「処理が順に流れること(フロー)」と「ある時点の値が正しいこと(状態)」は別の関心。フローの検証で状態の正しさまで背負わせない。状態は状態の階層が、フローはフローの階層が保証する。

### なぜ効くか
- **各階層が独立して機能する**:ある関心を検証する階層は、他階層が保証する事柄を前提にしてよい。例えば「入力値が受入仕様に妥当か」だけを見るテストは、描画も変数生成も他階層に委ね、自分はその一点に集中する。
- **変更の影響を局限できる**:受入仕様が変わってもその関心の階層だけが赤くなり、フローや描画のテストは巻き込まれない。脆さの主因(無関係な変更での赤)が構造的に消える。
- 結果としてフィードバックが速く、並行して触れる。これが「テストを階層化すると開発の機動力が上がる」の中身。

### 遂行手順(着手→完了)

階層化は「1枚に詰まった関心を、合否を独立に言える単位へ割り直す」作業だ。対象テスト(または設計しようとしている検証)に対して次の順で進める。

1. **そのテストが今見ている関心を全部書き出す**:1本のテストの中で何を確かめているかを列挙する(例: URL の描画 / 入力検証 / 変数割り当て)。2つ以上出たら、それは混在の信号。
2. **各関心を3つの軸で切れるか試す**:列挙した関心を「完結できる単位か」「別の境界付けられたコンテキストか」「フローか状態か」で振り分ける。隣の関心を前提にしないと合否を言えないものは、まだ割り切れていない。
3. **関心ごとに別階層へ分ける**:振り分けた関心を、それぞれ独立に合否判定できる別テスト(階層)にする。各階層は自分の関心だけを見て、他階層が保証する事柄は前提にしてよい。
4. **粒度の軸と掛け合わせる**:割った各関心に、配分論で決めた粒度(単体/結合/E2E)を掛ける(例: 単体レベル × 状態の関心)。関心の軸と粒度の軸は別物なので両方を指定する。
5. **増やしすぎを戻す**:文脈が1つしか無い関心まで階層に割っていないか確認し、過分割は1階層へ戻す(YAGNI)。階層はプロセスに要求されるコンテキストの有無で増減させる。

### 完了チェック(もれ確認)

- **各階層が単独で合否を言えるか**:どの階層のテストも、他階層のテストを実行せずに緑赤を判定できるか。隣を前提にしないと判定できなければ、まだ関心が混ざっている。
- **1テスト1関心になっているか**:手順1で2つ以上の関心が出たテストが、関心ごとに分割されたか。1枚に複数関心が残っていれば、無関係な変更で巻き込み赤化する。
- **フローと状態が分離したか**:フローを見るテストが状態の正しさまでアサートしていないか(逆も同様)。混在していれば片方の変更で他方が落ちる。
- **過分割していないか**:文脈が1つしか無いのに関心で階層を増やしていないか。増やした階層それぞれに固有のコンテキストがあることを確認する。
- **両軸を指定したか**:各テストに「粒度(単体/結合/E2E)」と「関心(フロー/状態/コンテキスト)」の両方が割り当たっているか。片方しか無ければ軸の取り違え。

### 落とし穴
- ツールが謳う区分(「これは E2E ツールだからフローも状態も全部見る」)に引きずられ、一枚に関心を詰め込む。ツールの用途でなく、関心事で割る。
- コンテキストが要らない場面まで階層を増やす。階層はプロセスに要求されるコンテキストの有無で増減させる。文脈が一つしか無いなら粒度の軸だけで足りる(YAGNI)。
- 粒度の軸と取り違える。これは「どこまで広く」ではなく「何の関心を」の軸。両軸は掛け合わせて使う(例: 単体レベル × 状態の関心)。

## リスクベーステスト(何から作るか)

抽出した振る舞いを全部いっぺんに同じ厚みでは作れない。
どれから手厚くテストするかを、**欠陥が出る確率 × 出たときの影響**で並べて決めるのがリスクベーステストである。

### 概要・目的

各振る舞いに、欠陥が混入しやすさ(複雑な分岐、頻繁に変わる箇所、新規・未知の技術、過去にバグが出た箇所)と、壊れたときの影響(金額、データ損失、法令、利用者数、復旧コスト)の2軸でスコアを振り、積の大きい順に優先する。
高リスクには手厚い網羅(境界・異常系・mutation・複数レベル)を、低リスクには代表1本を、という濃淡を意図的につける。
「全部を均一にテストする」は時間予算の前で必ず破綻し、結果として高リスクも低リスクも中途半端になる。先に並べて、上から埋める。

これは 0段の抽出台帳(SKILL.md)の優先度欄と直結する。
台帳の各 `T-ID` の優先度は、この確率 × 影響で付ける。優先度を落として「対象外」にするときは、低リスク(低確率かつ低影響)であることが落とす根拠になり、その1行を台帳に残す。

### 遂行手順(着手→完了)

リスクベースの本体は「全 `T-ID` をスコアで一列に並べ、上から手厚さを配る」ことだ。次の順で進める。頭の中で順位を付けず、台帳に数値で残す。

1. **採点軸を確定する**:確率(欠陥の混入しやすさ)と影響(壊れたときの損害)をそれぞれ 1〜3 の3段階で定義する。確率は「複雑な分岐・頻繁に変わる・新規技術・過去バグ」の該当数で、影響は「金額・データ損失・法令・利用者数・復旧コスト」の大きさで決める。段階の意味を先に1行で書き、ぶれを防ぐ。
2. **全 `T-ID` に確率スコアと影響スコアを付ける**:台帳の各行に確率(1〜3)と影響(1〜3)を1件ずつ振る。1つも飛ばさない(飛ばした行は順位が付かず、埋め忘れになる)。
3. **積でリスク値を出し、降順に並べる**:各行で `確率 × 影響`(1〜9)を計算し、その値で `T-ID` を降順ソートする。同点は影響の大きい方を上にする(低確率でも壊滅的な影響を埋もれさせない)。
4. **上位から手厚さを割り当てる**:リスク値の高い `T-ID` には手厚い網羅(境界・異常系・mutation・複数レベル)を、低い `T-ID` には代表1本を割り当てる。割り当てた厚さは配分論(上)のレベル割り当てと整合させる。
5. **台帳の優先度欄へ転記する**:出したリスク値と割り当てた厚さを、台帳の優先度欄へ書き戻す。低リスク(低確率かつ低影響)を理由に「対象外」へ落とすなら、その根拠を1行添えて残す(SKILL.md「やらないこと」)。
6. **変更のたびに見直す**:新機能投入・過去バグ再発領域など、リスクは動く。スコアと並びを再計算し、優先度欄を更新する。一度きりにしない。

### 完了チェック(もれ確認)

- **全 `T-ID` に確率と影響のスコアが付いたか**:台帳に確率・影響の両方が空欄なく入っているか(`grep -E '^\| T-' <台帳>` で両スコア列の空きを目視)。片方でも欠ければ順位が付かない。
- **両軸の積で並んでいるか**:確率だけ・影響だけで順位を付けていないか。低確率・高影響(資金移動、認証)が下位に沈んでいたら積で見直す。
- **上位ほど手厚いか**:リスク値の高い `T-ID` に手厚い網羅、低い `T-ID` に代表1本、の濃淡が付いているか。高リスクが代表1本どまりなら配分が逆。
- **対象外に根拠があるか**:優先度を落として外した `T-ID` に「低確率かつ低影響」の根拠が1行残っているか。異常系・境界を理由なく対象外にしていないか(高リスクの異常系は最優先)。
- **再評価の痕跡があるか**:直近の変更(新機能・バグ修正)に対してスコアを見直したか。一度きりの評価のまま放置していないか。

### 落とし穴

- 確率だけ、影響だけで決める。低確率でも影響が壊滅的(資金移動、認証)なら上位に来る。両軸の積で見る。
- リスク評価を一度きりにする。リスクは変わる(新機能投入、過去バグの再発領域)。台帳の優先度は変更のたびに見直す。
- 「低リスクだから書かない」と「異常系だから対象外」を混同する。異常系でも高リスクなら最優先。安易な対象外送りの口実にしない(SKILL.md「やらないこと」)。
