# EARS 化の補助パターン集

`SKILL.md` の基本5パターンだけでは足りない局面向け。EARS 化前のトリアージ、基本5パターンを跨ぐ複合表現、
書き上げた EARS 文のセルフレビュー観点の3つを収める。理論名・カテゴリ名はここに閉じ、台帳や本体へは
変換後の EARS 文だけを書く。

## 1. EARS 化前の弱点トリアージ

元の要求文(自然言語)を1文ずつ、そのまま EARS へ変換する前に次の弱点が無いか確認する。
ある場合、変換先の EARS 文でその弱点を埋める(トリガー・条件・数値を補う)まで変換完了としない。

- **広すぎる**: 「認証機能を追加」→ パスワード要件、セッション管理が抜けていないか
- **トリガー欠落**: 「通知を送る」→ いつ/何をきっかけに送るかが無い
- **動作があいまい**: 「使いやすくする」→ 数値かユーザー観測可能な条件に落とせるか
- **制約が無い**: 「決済を処理する」→ セキュリティ・コンプライアンス上の制約が書かれていない

0 段([`../../loopeng-extract/SKILL.md`](../../loopeng-extract/SKILL.md))の抽出台帳の行ごとに当てるチェック。

対象ドメイン(時間管理、認証、決済、多段階フォーム等)に典型的な要件カテゴリ(締切上限、失敗回数閾値、
二重処理防止、途中離脱時の保存 等)を一通り思い浮かべ、台帳に該当 S-xxx 行が無ければ抽出漏れを疑う。

## 2. 基本5パターンを跨ぐ複合表現

- **多条件イベント**: `When <trigger> AND <condition2>, the system shall <action>.`
- **ロールベース権限**: アクターを主語に固定して1ロール1文に割る(1文に複数ロールの分岐を書かない)。
  `If user role is "Admin", the system shall <action>.` / `If user role is "Viewer", the system shall prevent <action>.`
- **優先順位付き分岐**: 複数の unwanted 条件が同時成立しうる場合、順位を明示する(暗黙に決めない)。
  `Priority 1: If <A>, the system shall <action-A>. Priority 2: Else if <B>, the system shall <action-B>. Default: <fallback>.`
- **定量化した NFR**: 「速い」「安全」は不可、数値・パーセンタイル・単位を伴わせる。
  `When a search query is submitted, the system shall return results within 200ms for 95% of requests.`

いずれも TLA+ へは `../../loopeng-formalize/SKILL.md` の「TLA+ spec への落とし込み」節の手順(1節=1
disjunct、優先順位は guard 条件に埋め込む)でそのまま渡せる。

## 3. EARS 文のセルフレビュー観点(落とし穴)

書き上げた EARS 文を台帳に `[x]` する前に確認する。見つかったら `[ ]` に戻して直す。

- **循環条件**: 「A なら B、B なら A」の閉路になっていないか
- **到達不能分岐**: 型域・状態遷移上あり得ない条件を書いていないか
- **優先順位の欠落**: 複数条件が同時成立しうるのに勝敗を書いていないか
- **デフォルトケースの欠如**: else/otherwise/fallback が無いか
- **あいまいな閾値**: 「大量」「頻繁」を数値化せずに残していないか
- **異常系ハンドリングの欠如**: 外部依存の失敗時挙動が書かれているか
