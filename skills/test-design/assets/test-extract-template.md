# テスト項目抽出台帳: <対象名>

対象: <機能名 / モジュール / API / PR の差分範囲>
走査アンカー: <受け入れ条件 / 関数シグネチャ / 分岐 / 状態遷移 / API エンドポイント×メソッド / 不変条件>

各 ID は「テストすべき振る舞い」1つ。正常系、境界、異常系、非機能を網羅的に出し切る。
手法(どの reference の技法で検証するか)を割り当て、テスト名へ落とし終えたら「設計済」を `[x]`。
実行ゲート(緑、型、flaky なし)を通ったら「緑確認済」を `[x]`。
「設計済」が1つでも残る、または連番に欠番があるうちは抽出は未完。「緑確認済」が全行 `[x]` で初めて完了。
原則は **過剰抽出は安全、漏れは危険**。迷ったら採る(後で「優先度低/対象外」に落とせる)。

## 用語定義(抽出より先に埋める)

対象に出てくる曖昧な語を一つずつ確定させてから振る舞いを出す。
語の定義が曖昧なまま進めると、境界(連休、月末、空、上限)と異常系がそっくり抜ける。
**用語の未定義は最大の抽出漏れ源**。暗黙の「当たり前要求」(セキュリティ、実行効率、法令)もここで明示に引き上げる。

| 用語 | 定義(曖昧さを残さない) | この定義が生む境界と異常系 |
|------|--------------------------|----------------------------|
| <例: 営業日> | <休日でない日。休日=日曜、指定土曜、指定祝日> | <連休跨ぎ、月末→翌月、年末年始> |
|      |                          |                            |

## 台帳

- [ ] T-001 <出典: 受け入れ条件/行/分岐> 「<対象の振る舞い>」
      → 手法: <同値分割 (blackbox-partition.md) / 境界値 / PBT ...> / テスト名: <test_*>
- [ ] T-002 <出典> 「<振る舞い>」
      → 手法: <手法名 (reference ファイル名)> / テスト名: <...>

手法欄には参照した reference ファイル名を併記する(reference を開いて落とし穴まで読んだ証跡)。

## トレーサビリティマトリクス(消し込み台帳)

漏れが「空欄」として見える形にする。
「振る舞いはあるのに手法が空」= 設計漏れ。「手法はあるのにテストが空」= 実装漏れ。

「設計済」= 手法とテスト名が埋まった。「緑確認済」= 実行ゲートを通った(緑、型クリーン、flaky なし)。
**緑確認済が全行 `[x]` になって初めて抽出を閉じる**(設計済だけでは閉じない)。critical な行(金額、セキュリティ、状態遷移)は mutation 列も埋める。

| 振る舞い(T番号) | 種別(正常/境界/異常/非機能) | 手法(reference) | テスト名 | 設計済 | 緑確認済 | mutation(criticalのみ) |
|------------------|------------------------------|------------------|----------|--------|----------|------------------------|
| T-001            |                              |                  |          | [ ]    | [ ]      | —                      |
| T-002            |                              |                  |          | [ ]    | [ ]      | —                      |

## 網羅を底上げする2手(任意だが効く)

- **unwanted の系統生成**：各正常系に「この入力が不正なら? 境界を踏んだら? 並行で起きたら?」を必ず問い、異常系の行を立てる。
- **多視点の独立抽出**：仕様視点(ブラックボックス)とコード視点(ホワイトボックス/分岐)で独立に出し、union を取る。観点が違うと拾うものが違う。

## 閉じる前のチェック(設計用: 0段を閉じる条件)

- [ ] 用語定義表が埋まり、曖昧な語が残っていない
- [ ] 全 T-ID の「設計済」が `[x]`(未チェックなし)
- [ ] T-001 から連番に欠番なし
- [ ] 各 T-ID に手法が割り当たっている(マトリクスの「手法」列に空欄なし、reference ファイル名つき)
- [ ] 正常系だけでなく、境界、異常系、(必要なら)非機能が各機能に最低1つある

## 緑にする前のチェック(実行ゲート: 自己申告を信用しない)

設計完了では閉じない。下記を生ログで自分で確認して初めて「緑確認済」を `[x]` にする。

- [ ] 全 T-ID について、対応するテスト名が実在のテストファイル内に見つかる(grep で逆引き突合。委譲先が取りこぼした行が無いか)
- [ ] 全テストが緑(テストランナーの生ログを確認。報告だけで信じない)
- [ ] 揺らぎを含むテスト(PBT/メタモルフィック/並行)は seed を変え複数回回しても緑
- [ ] 型検査がクリーン
- [ ] critical な T-ID(金額/セキュリティ/状態遷移)は mutation を回し、survived がアサーション不足として残っていない

## レビュー用途のチェック(既存テストの抜け点検)

両方向を機械的に回す。読んで分かった気にならない。

- [ ] 既存テストを**全件**台帳右側へ転記した(テストファイルから抽出した件数と右側の件数が一致する)
- [ ] 右→左:右側の各テストに対応する左側 T-ID がある(無い=根拠不明テスト。抽出漏れか脆さを疑う)
- [ ] 左→右:左側で右側が空の T-ID を列挙した(=テストの抜け)
- [ ] 右側に埋めたテストが skip/pending でなく現に緑で走っている(skip 黙殺は守られていないテスト)
- [ ] 手法が実装詳細に密結合したテストを洗い出した(=脆さ。test-design-quality.md 参照)
