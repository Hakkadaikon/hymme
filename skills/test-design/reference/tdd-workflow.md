# TDD の進め方

テスト駆動開発で実装を駆動する手順と、Green に至る戦略をまとめる。
失敗するテストを起点に、駆動のサイクルを小さく回す側の規範である。
出典は t-wada(和田卓人)の TDD 観点。

「良いテストとは何か」の規範は [`good-test-principles.md`](good-test-principles.md)、テストダブルの使い分けは [`test-doubles.md`](test-doubles.md) を参照。

各項目は「概要」「目的/いつ使う」「TypeScript example(vitest 想定)」「落とし穴」の構成で示す。

## TDD(Red→Green→Refactor)

### 概要
失敗するテストを先に書き(**Red**)、それを通す最小のコードを書き(**Green**)、テストを保ったまま設計を整える(**Refactor**)、という三拍子を小さく回す開発手法。

### 目的/いつ使う
実装を外側の振る舞いから駆動し、テスト不能な設計に流れるのを防ぐ。
非自明なロジック(分岐、ループ、パーサー、金額やセキュリティの経路)で使う。
設定値や自明な一行に持ち込むのは過剰で、YAGNI に反する。

各フェーズの規律は次のとおり。

- **Red**：テストを一本だけ書き、実際に実行して落ちることを確認する。落ちる理由が期待どおりかも見る。これはテスト自身のバグを検出する工程でもある。
- **Green**：そのテストを通す最小限のコードを書く。ベタ書きでよい。まず緑にすることを優先する。
- **Refactor**：緑を保ったまま重複を除き、設計を整える。テストが通り続けることを各ステップで確認する。手順と勘所は独立節[Refactor フェーズ(緑のまま設計を整える)](#refactor-フェーズ緑のまま設計を整える)に展開する。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";

// Red: まだ存在しない関数を呼ぶテストを書き、実行して落ちることを確認する
import { fizzbuzz } from "./fizzbuzz";

describe("fizzbuzz", () => {
  it("returns 'Fizz' for multiples of 3", () => {
    expect(fizzbuzz(3)).toBe("Fizz");
  });
});

// Green: 最小実装。この時点ではベタ書きでよい
// export const fizzbuzz = (n: number): string => (n % 3 === 0 ? "Fizz" : String(n));
```

### 落とし穴
- Red を飛ばして実装から書くと、テストが常に緑のまま壊れていても気付けない。落ちることの確認を省かない。
- Green で先回りして一般化すると、テストに支えられていないコードが生まれる。最小実装にとどめる。

### 網羅とのつなぎ方
網羅は、テストリストの全項目を一つずつ Red から Green へ消化し切ったときに達成される。
この三拍子の手順そのものは網羅基準を生まない。
何を網羅すべきかは、別途ブラックボックス技法(同値分割、境界値)やホワイトボックス技法(分岐網羅)で定める。
それらが出した項目集合を、本ループが一項目ずつ潰していく実行機構になる。
Red を飛ばすと、緑のまま壊れていても気付けない網羅、つまり保護にならない網羅が積み上がる。
落ちることの確認は、各項目が「本当にその振る舞いを押さえている」ことの担保になる。
網羅の停止条件は「リストが空になった」であり、手順の側ではなくリストの側にある。

## Refactor フェーズ(緑のまま設計を整える)

### 概要
緑になったテストを安全網として、振る舞いを変えずに内部構造だけを整える工程。
Red と Green が「動くものを作る」のに対し、Refactor は「動いたものをきれいにする」側で、三拍子の最後に必ず置く。

### 目的/いつ使う
仮実装や三角測量で緑にした直後、重複やベタ書きの分岐が残っているときに使う。
Green の勢いで書いたコードは、しばしば重複と場当たりの分岐を抱える。
それを次の Red に進む前に解消し、設計の負債を持ち越さない。

#### リファクタの安全網としてのテスト
緑のテストが現在の振る舞いを固定しているからこそ、内部を安心して作り替えられる。
リファクタとは「テストを緑に保ったまま」内部を変える操作であり、テストが赤に振れたら、それは振る舞いを壊した合図になる。
だから**テストを変えながらリファクタしない**。
テスト側とプロダクション側を同時に動かすと、どちらが壊れたのか切り分けられず、安全網が機能しない。
テストの構造自体を直したいときは、プロダクションコードのリファクタとは別サイクルに分け、一度に一つだけ動かす。

#### 緑のまま小さく刻む手順
一回のリファクタは小さく刻み、各ステップで緑を確認しながら進める。

1. リファクタを一つだけ行う(重複を一箇所くくり出す、変数を一つ改名する、分岐を一段ほどく)。
2. テストを実行する。
3. 緑が維持されていることを確認する。緑なら次の一手へ。
4. 赤になったら、その一手を即座に戻して緑へ復帰する。原因を分析してから、より小さい一手に割り直す。

赤を抱えたまま次の変更を重ねない。
赤の上に積むと、どの一手が壊したのかが埋もれ、戻す先を失う。

#### 重複除去のタイミング(Rule of Three)
重複は見つけ次第すべて消すのではなく、**三度目の出現で除去する**(Rule of Three)。
一度目は書く。二度目は重複に気づきつつ許容する。三度目で初めて共通化する。
早すぎる共通化は、まだ揃っていない構造を無理にまとめ、後で剥がす負債になる。

これは仮実装・三角測量と噛み合う。
仮実装で定数を返し、三角測量で二本目・三本目のテストを足すと、各ケースに似た処理が並ぶ。
その重複が三度目で揃ったところを Refactor で本実装に寄せると、定数返しが一般化されたロジックへ昇格する。
Green で一般化を先回りせず、Refactor で重複を根拠に一般化する、という分業になる。

### TypeScript example
三角測量で緑にした直後、各テストに重複したセットアップが並んでいる。
これを緑のまま共通化する。

```ts
import { describe, it, expect, beforeEach } from "vitest";
import { Cart } from "./cart";

// before: 各 it が同じ new Cart + addItem を繰り返している(重複が三度出ている)
describe("Cart 合計", () => {
  it("単一商品の合計", () => {
    const cart = new Cart();
    cart.addItem({ price: 100, qty: 2 });
    expect(cart.total()).toBe(200);
  });
  it("複数商品の合計", () => {
    const cart = new Cart();
    cart.addItem({ price: 100, qty: 2 });
    cart.addItem({ price: 50, qty: 1 });
    expect(cart.total()).toBe(250);
  });
});

// after: 重複したセットアップを beforeEach に抽出。振る舞いは不変、テストは緑のまま
describe("Cart 合計", () => {
  let cart: Cart;
  beforeEach(() => {
    cart = new Cart();
    cart.addItem({ price: 100, qty: 2 });
  });
  it("単一商品の合計", () => {
    expect(cart.total()).toBe(200);
  });
  it("複数商品の合計", () => {
    cart.addItem({ price: 50, qty: 1 });
    expect(cart.total()).toBe(250);
  });
});
```

抽出は一度に一つ、抽出のたびにテストを走らせて緑を確認する。
緑が崩れたら抽出を戻し、より小さい単位でやり直す。

### 落とし穴
- テストを変えながらプロダクションコードをリファクタし、安全網を自分で外す。一度に一つだけ動かす。
- 一度目の重複で共通化を急ぎ、揃っていない構造を無理にまとめる。三度目まで待つ(Rule of Three)。
- 赤を抱えたまま次のリファクタを重ね、戻す先を失う。赤になったら即座に戻す。
- 「リファクタ」と称して振る舞いを変える(挙動の修正や機能追加を混ぜる)。それは Refactor ではなく次の Red から始める別サイクル。

### 網羅とのつなぎ方
Refactor は網羅範囲を増やさない。
緑の集合を保ったまま内部を整える工程であり、新しい振る舞いを足すなら次の Red に戻る。
網羅を進めるのは Red→Green であって、Refactor はその成果を壊さず維持する役割に徹する。
ただし重複除去で本実装に寄せた一般化は、同値クラス全体を一例で代表できる形に近づけ、後続の網羅効率を上げる。

## テストリスト先行

### 概要
着手前に、実装すべき振る舞いを箇条書きのテストリストへ洗い出す。
t-wada 流 TDD の起点であり、正常系、境界、異常系を網羅的に並べる。

### 目的/いつ使う
何を確かめるべきかを先に決め、行き当たりばったりの実装を防ぐ。
リストは一つずつ消化し、作業中に思いついた項目は随時追記する。
全項目を最初に完璧へ揃える必要はなく、見えているものから書き出して走り出す。

### TypeScript example
リストはコードではなく、計画として書く。
```text
# uint パーサー parseUint のテストリスト
- [ ] "0" -> 0
- [ ] "42" -> 42
- [ ] 先頭ゼロ "007" -> 7      (正規化の境界)
- [ ] "" -> エラー            (異常系: 空)
- [ ] "-1" -> エラー          (異常系: 符号)
- [ ] "1.5" -> エラー         (異常系: 非整数)
- [ ] 巨大値の上限           (境界: オーバーフロー)
```

### 落とし穴
- 正常系だけを並べ、異常系と境界を書き落とす。網羅の漏れがそのまま品質の漏れになる。
- リストを清書して固定文書にしようとする。リストは作業用のメモであり、消化と追記で動き続けてよい。

### 網羅とのつなぎ方
テストリスト先行は、網羅の入口である。
リストの網羅性が、そのままテストの網羅性になる。
だからリストを書く段階で、正常系、境界、異常系を体系的技法で出し切る。
代表値は同値分割で、端点はオフバイワンを狙う境界値分析で、状態を持つなら状態遷移で導く。
思いつきで並べると正常系に偏り、その偏りが網羅の穴としてそのまま残る。
出し切った項目をリスト化し、全項目にチェックが付いた時点を網羅完了とみなす。
作業中に見つけた抜けは、その場でリストへ追記して網羅対象に組み込む。
リストは消化と追記で動き続け、その動きを通じて網羅の範囲が確定していく。

## 三角測量(Triangulation)

### 概要
一般化の方針が見えないとき、二つ目、三つ目の具体例(テスト)を足して、仮実装を本実装へ追い込む技法。

### 目的/いつ使う
最初のテストをハードコードで通したあと、一般解への道筋が自明でないときに使う。
方針が自明なら、わざわざ例を増やさず「明白な実装」で一気に書いてよい。
三角測量はあくまで方針が見えないときの補助輪である。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";
import { add } from "./add";

describe("add: triangulation", () => {
  // 1本目だけなら `return 3` でも通る(仮実装)
  it("1 + 2 = 3", () => expect(add(1, 2)).toBe(3));
  // 2本目を足すと、定数では通らず一般化を強制される
  it("3 + 4 = 7", () => expect(add(3, 4)).toBe(7));
});
```

### 落とし穴
- 方針が自明な箇所でも機械的に二本目を足し、冗長なテストを量産する。
- 二本目が一本目と同じ同値クラスにあり、一般化を強制できていない。異なる代表値を選ぶ。

### 網羅とのつなぎ方
網羅の観点では、三角測量は異なる同値クラスの代表で複数例を足し、一般化を強制する操作である。
二本目を一本目と同じ同値クラスから取ると、定数返しのまま通り、網羅は一歩も進まない。
だから二本目以降は、未踏の同値クラスや境界の端点から選ぶ。
こうして足した例の一つひとつが、リスト上の別項目を消化する。
一般化が完成すれば、その同値クラス全体を一例で代表できたことになり、網羅効率が上がる。
方針が自明な箇所では、例を増やさず明白な実装で書けばよく、機械的な二本目は冗長な網羅になる(YAGNI)。

## 仮実装と明白な実装(Fake it / Obvious implementation)

### 概要
Green に至る二つの戦略。
**仮実装**は定数を返すなどベタ書きでまず緑にする。
**明白な実装**は正解が自明なときに本実装を直接書く。

### 目的/いつ使う
不安が大きい、または一般化の形が見えないときは仮実装から入り、三角測量で一般化する。
正解に確信があり書き間違える恐れが小さいときは、明白な実装で一気に書く。
二つは対立せず、確信度に応じて選ぶ。

### TypeScript example
```ts
// 仮実装: まず定数で緑にし、後で一般化する
export const square = (n: number): number => 4; // square(2) のテストだけなら通る

// 明白な実装: 正解が自明なので直接書く
export const squareObvious = (n: number): number => n * n;
```

### 落とし穴
- 確信もないのに明白な実装へ飛び、緑だが間違ったコードを書く。不安なら仮実装に戻る。
- 仮実装のまま三角測量を忘れ、定数返しが残る。一般化まで進めて初めて完了する。

### 網羅とのつなぎ方
仮実装か明白な実装かの選択は、網羅とは直交する Green の戦略であり、確信度で選ぶ。
網羅範囲を決めるのはリストであって、どちらの戦略で緑にしたかではない。
ただし仮実装のまま三角測量を忘れると、リスト項目を網羅したつもりで定数返しが残る危険がある。
このとき緑のチェックは付くが、実装は一般化されておらず、網羅は見かけだけになる。
網羅が成立するのは、各リスト項目が一般化された実装の上で緑になったときである。
明白な実装で一気に書く箇所は、その一歩で対応するリスト項目が一般化込みで埋まる。
