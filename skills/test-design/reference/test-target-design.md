# テスト対象の設計と配置

テストしやすい形へ対象を設計し、どこに労力を配るかを決める側面をまとめる。
出典は [`good-test-principles.md`](good-test-principles.md) と同じく Vladimir Khorikov『単体テストの考え方/使い方』。

脆さの原因を断つ側面(脆さの回避、モック濫用の回避)は [`test-design-quality.md`](test-design-quality.md) に置く。
こちらは、テストしやすい形へ設計を割り(Humble Object)、労力の配分を決め(四象限)、テスト項目の起点を定める(契約による境界)、テスト**対象側の設計**を扱う。

各項目は「概要」「目的/いつ使う」「TypeScript example(vitest 想定)」「落とし穴」「網羅設計での効かせ方」の構成で示す。

## Humble Object パターン

### 概要
テストしにくい層(UI、データベース、外部 I/O)から、テストしやすいロジックを引き剥がす設計。
テスト困難な層は、ロジックを持たない薄い「控えめな(humble)」殻にする。

### 目的/いつ使う
ロジックが I/O やフレームワークと絡んでテストできないときに使う。
判断や計算をプレーンな関数やドメインモデルへ抜き出し、出力ベースで単体テストする。
殻に残るのは委譲だけなので、そこは単体テストの対象から外し、必要なら少数の結合テストで覆う。
これは四象限でいう「過度に複雑なコード」を、ドメインモデルとコントローラへ割るための主要な道具である。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";

// 抜き出したロジック(テスト容易・純粋)
export const nextRetryDelay = (attempt: number): number =>
  Math.min(1000 * 2 ** attempt, 30_000);

// 控えめな殻(I/O のみ。ロジックを持たないので単体テスト対象外)
// async function retryLoop(task) { await sleep(nextRetryDelay(n)); ... }

it("caps the retry delay", () => {
  expect(nextRetryDelay(0)).toBe(1000);
  expect(nextRetryDelay(10)).toBe(30_000); // 上限に張り付く境界
});
```

### 落とし穴
- 殻にロジックが残ったまま分離した気になり、結局テストできない判断が I/O 層に居座る。
- 抜き出した関数が外部状態へ依存し、純粋でなくなる。入力は引数で受け、出力は戻り値で返す。

### 網羅設計での効かせ方
テスト困難で網羅できない層からは、まず網羅対象のロジックを純粋関数へ引き剥がす。
I/O やフレームワークと絡んだままでは、分岐や境界を網羅しようにも準備が高くつく。
判断や計算を純粋関数へ抜き出せば、入力と戻り値の組で出力ベースに厚く網羅できる。
残った殻は委譲だけなので網羅対象から外し、必要なら少数の結合テストで覆う。
こうして「網羅したいロジック」と「網羅しない殻」を分け、労力を前者へ集める。
網羅の達成チェックは、抜き出した純粋関数の側で分岐と境界が踏めているかで行う。
殻に判断が残っていれば、それは網羅対象が殻へ漏れている信号として扱う。

## テスト対象の四象限

### 概要
コードを、ドメインの複雑さと協力者への依存の二軸で四つに分ける。

- **ドメインモデルとアルゴリズム**(複雑さ高、依存少)：単体テストの費用対効果が最も高い。最優先で厚く書く。
- **取るに足らないコード**(複雑さ低、依存少)：単純な getter など。テスト不要。
- **コントローラ**(複雑さ低、依存多)：依存を束ねて流すだけの層。少数の結合テストで覆う。
- **過度に複雑なコード**(複雑さ高、依存多)：最も危険。テストが高くつき、しかも書かないと事故る。

### 目的/いつ使う
どこにテストの労力を注ぐかを決める地図に使う。
過度に複雑なコードは、そのまま正面からテストしようとせず、Humble Object でドメインモデル(複雑さ高、依存少)とコントローラ(複雑さ低、依存多)へ割る。
割ったあとは、ドメインモデルを出力ベースで厚くテストし、コントローラは結合テストへ回す。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";

// ドメインモデル(複雑さ高・依存少): 出力ベースで厚くテストする価値が最も高い
export const isLeapYear = (y: number): boolean =>
  y % 4 === 0 && (y % 100 !== 0 || y % 400 === 0);

describe("isLeapYear: domain model gets the most tests", () => {
  it.each([
    [2000, true],  // 400 で割れる
    [1900, false], // 100 で割れるが 400 では割れない(境界)
    [2024, true],
    [2023, false],
  ])("%i -> %s", (y, expected) => expect(isLeapYear(y)).toBe(expected));
});
```

### 落とし穴
- 取るに足らないコードに丁寧なテストを書き、保守コストだけ増やす。
- 過度に複雑なコードを分割せず正面からテストし、脆く高価なテストを抱える。

### 網羅設計での効かせ方
四象限は、網羅の労力をどこへ厚く配分するかを示す地図として使う。
ドメインモデルとアルゴリズム(複雑さ高、依存少)に、網羅を最も厚く取る。
ここは分岐と境界が多く依存が少ないので、出力ベースで高い網羅率を安く達成できる。
取るに足らないコード(複雑さ低、依存少)は、網羅対象から外してよい。
コントローラ(複雑さ低、依存多)は、単体で網羅せず少数の結合テストで薄く覆う。
過度に複雑なコード(複雑さ高、依存多)は、正面から網羅しようとすると脆く高くつく。
これは Humble Object でドメインモデルとコントローラへ割り、割った後でドメイン側を厚く網羅する。
網羅率の目標は象限ごとに変え、一律の数字を全コードへ課さないのが要点だ。

## 契約による境界(事前条件・事後条件)

### 概要
対象の責務を**事前条件**(呼び出し側が満たす義務)と**事後条件**(満たされたとき対象が保証する出力・状態変化)に分けて捉える。
テストの起点を「入力をどう散らすか」でなく「契約の境界はどこか」に置く考え方(DbC)。

### 目的/いつ使う
関数・API・モジュールのテスト項目を、何を保証し何を保証しないかで仕分けたいとき。
特に「不正入力に対する振る舞い」を異常系として書くべきか迷ったときの判断基準になる。

- **事前条件を満たさない入力は、原則テスト対象にしない。** 契約上、対象はどう振る舞ってもよい(クラッシュ含む)から、特定の振る舞いを期待するテストは過剰仕様になり実装を不当に縛る。
- **不正入力に定まった応答を保証したいなら、それは事前条件ではなく事後条件である。** 例: REST で不正ボディに 400 を返すことを保証したいなら、「不正ボディは事前条件違反」ではなく「あらゆるボディを受け、不正なら 400 を返す」という事後条件に書き換える。そう書いた瞬間、それは正常系の網羅対象になる。
- 事後条件は入力区分ごとに出力を表で明記し、各行を1テストに落とす(`blackbox-systematic.md` のデシジョンテーブル・同値分割と同じ向き)。

### TypeScript example
```ts
// 事前条件: n は非負整数。満たさない入力(負数, 小数, NaN)は契約外 → テストしない。
function fizzbuzz(n: number): string { /* ... */ }

describe("fizzbuzz (事後条件: 入力区分ごとの出力)", () => {
  it.each([
    [3, "Fizz"], [5, "Buzz"], [15, "FizzBuzz"], [1, "1"],
  ])("f(%i) = %s", (n, want) => expect(fizzbuzz(n)).toBe(want));
  // f(-1) や f(2.5) のテストは書かない。期待する振る舞いが契約上存在しないため。
});

// 一方 API は「不正入力に 400」を保証したい → 事後条件に引き上げる。
describe("POST /users (事後条件: 不正ボディは 400)", () => {
  it("rejects malformed body with 400", async () => {
    expect((await post("/users", { name: "" })).status).toBe(400);
  });
});
```

### 落とし穴
- 事前条件違反のテストを大量に書いて、実装を契約より狭く縛る。後で正当な実装変更がテストで落ちる。
- 逆に、保証したい不正入力応答を「事前条件違反だから」とテストせず放置する。**保証したいなら事後条件に書く**。この一手を飛ばすと境界の保証が穴になる。
- 入力検証(trust boundary)は契約の外に出さない。外部入力の検証は常に事後条件として明示する(ponytail でも省かない領域)。

### 網羅設計での効かせ方
0段の抽出(SKILL.md)で各振る舞いに種別を振るとき、まず事前条件を確定させる。
事前条件の内側だけが正常系・境界の網羅対象、事前条件違反のうち**応答を保証する分だけ**を事後条件へ引き上げて異常系に積む。
これで「不正入力ケースを無限に増やす」のと「保証すべき異常系を取りこぼす」の両方を同時に避けられる。
