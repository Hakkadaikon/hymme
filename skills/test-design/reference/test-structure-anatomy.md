# テストの構成パターン(内部構成)

個々のテスト一本を、本体の中でどう構成するかの型をまとめる。
テスト一本の内部を、準備、実行、検証の段に切り分ける配置の規範であり、どんな順序で実装を駆動するかとは別の話題である。
出典は t-wada(和田卓人)の TDD 観点。

このファイルはテスト本体の**内部構成**(段の切り方)を扱う。
リソースのライフサイクル(setup/teardown)とテストデータの供給(Builder / Object Mother)は [`test-structure-lifecycle.md`](test-structure-lifecycle.md) を参照。
「良いテストとは何か」の規範は [`good-test-principles.md`](good-test-principles.md)、テストダブルの使い分けは [`test-doubles.md`](test-doubles.md) を参照。

各項目は「概要」「目的/いつ使う」「TypeScript example(vitest 想定)」「落とし穴」「遂行手順」「完了チェック」「網羅とのつなぎ方」の構成で示す。

## AAA パターン(Arrange-Act-Assert)

### 概要
テスト本体を、準備(**Arrange**)、実行(**Act**)、検証(**Assert**)の三段に分ける構成。

### 目的/いつ使う
ほぼすべての単体テストで使える基本構成。
各段を空行で区切ると、何を準備し、何を実行し、何を確かめたかが一目で追える。
Act が複数行に膨らむのは、テスト対象の API が使いにくい兆候である。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";
import { Cart } from "./cart";

describe("Cart", () => {
  it("sums line totals", () => {
    // Arrange
    const cart = new Cart();
    cart.add({ price: 100, qty: 2 });

    // Act
    const total = cart.total();

    // Assert
    expect(total).toBe(200);
  });
});
```

### 落とし穴
- Assert が複数の無関係な事柄を一度に検証し、一テスト一振る舞いの原則を崩す。
- Arrange が肥大して、何の準備が結果に効くのか読み取れなくなる。準備の核だけを残す。

### 遂行手順(着手→完了)
AAA で1本書く作業の本体は「準備・実行・検証の境目に線を引き、Act を1操作・Assert を1関心へ絞る」ことだ。次の順で進める。

1. **検証する振る舞いを1つに固定する**:台帳の `T-ID` 1行を選び、そのテストが確かめる振る舞いを一文で言い切る(「2行を足すと合計が出る」)。言い切れないなら振る舞いが2つ混ざっているので、テストを分けるところから始める。
2. **Act を1行に決める**:テスト対象の「その振る舞いを起こす唯一の呼び出し」を先に書く(`const total = cart.total()`)。ここを起点にすると、上に必要な準備、下に必要な検証が自ずと決まる。Act が2行以上要るなら、API が使いにくい兆候として記録し、まず1呼び出しで済む形を疑う。
3. **Arrange に「結果に効く準備」だけ置く**:Act が成立する最小の前提だけを Act の上に書く。既定値で足りるフィールドは触らない(肥大の核を作らない)。準備が長くなるなら Test Data Builder へ逃がす([`test-structure-lifecycle.md`](test-structure-lifecycle.md) の Test Data Builder / Object Mother)。
4. **段の間に空行を1つずつ入れる**:Arrange / Act / Assert の3ブロックを空行で区切る。コメント `// Arrange` 等は補助で、区切りの本体は空行。3ブロックを超える空行群があれば段が増えている=振る舞いが複数。
5. **Assert を1関心に絞る**:手順1で言い切った振る舞いの結果だけをアサートする(`expect(total).toBe(200)`)。「ついでに」別の値を確かめたくなったら、それは別の `T-ID` の領分。
6. **失敗を1回見る**:アサートの期待値をわざと1つ崩して走らせ、狙った1行で落ちることを確認してから戻す(検証が実際に効いていることの確認)。

### 完了チェック(もれ確認)
- **段が3つに割れているか**:テスト本体の空行が Arrange/Act/Assert の2箇所だけで、4ブロック以上に割れていないか。割れていれば振る舞いが複数混入。
- **Act が1行か**:Act ブロックが単一の呼び出しか。複数行なら API の使いにくさか、振る舞いの混在を疑う。
- **Assert が1関心か**:`grep -n "expect(" <test>` のヒットが、すべて手順1で言い切った1つの振る舞いを指しているか。無関係な対象への `expect` が混じれば一テスト一振る舞いの違反。
- **Arrange に死んだ準備が無いか**:Arrange の各行を消してみて Act/Assert がなお意味を保つものは、結果に効かない飾り。核だけ残す。
- **落として確認したか**:手順6の「わざと壊して1行で落ちる」を実行済みか(常に緑のテストは何も守っていない)。

### 網羅とのつなぎ方
AAA は、一テスト一振る舞いを保ち、網羅対象である振る舞いとテストを一対一に対応づける構成である。
この一対一が崩れると、何本書いたかと何を網羅したかがずれ、網羅の勘定が合わなくなる。
Assert に無関係な検証を詰め込むと、一本のテストが複数の振る舞いを曖昧に跨ぎ、振る舞い単位の網羅が数えられなくなる。
だから検証は、そのテストが担う一つの振る舞いに絞る。
Act が複数行に膨らむのは、一つの操作で一つの振る舞いを呼べていない兆候であり、網羅対象の切り方を見直す合図になる。
この一対一を保つと、リスト項目とテストが素直に対応し、網羅の達成度をテスト本数で追える。

## Given-When-Then(BDD 由来)

### 概要
AAA と同型の構成を、振る舞い駆動開発(BDD)の語彙で表す。
前提(**Given**)、操作(**When**)、結果(**Then**)の三段からなる。

### 目的/いつ使う
非技術者にも読める言葉で仕様を記述したいときに使う。
受け入れテストや、Gherkin 形式のシナリオと相性がよい。
構造は AAA と同じであり、語彙が業務寄りになる点だけが違う。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";
import { applyDiscount } from "./discount";

describe("discount for members", () => {
  it("gives 10% off to members", () => {
    // Given a member and a 1000 yen item
    const member = { isMember: true };

    // When the discount is applied
    const price = applyDiscount(1000, member);

    // Then the price is reduced by 10%
    expect(price).toBe(900);
  });
});
```

### 落とし穴
- Given に業務上意味のない技術的詳細を詰め込み、シナリオとしての読みやすさを失う。
- When に操作を二つ以上入れ、どちらが結果を生んだか曖昧にする。

### 遂行手順(着手→完了)
GWT で1本書く作業の本体は「業務語彙で前提・操作・結果の三段を立て、技術詳細を Given から締め出す」ことだ。AAA と同型なので段の切り方は AAA に従い、ここでは語彙を業務へ寄せる点を足す。

1. **対応する受け入れシナリオを1つ選ぶ**:台帳の `T-ID`、または受け入れ条件/Gherkin シナリオ1件を選ぶ。GWT の網羅対象はシナリオの一覧なので、シナリオが先、テストが後。
2. **Then(結果)を業務の言葉で先に書く**:「会員価格になる」のように、確かめたい結果をまず1つ、業務が読める語で言い切る。Then を起点にすると Given/When が逆算で決まる。
3. **When を1操作に絞る**:その結果を生む唯一の業務操作を書く(`applyDiscount(1000, member)`)。操作を2つ入れると、どちらが Then を生んだか辿れなくなる。
4. **Given に業務前提だけ置く**:When が成立する業務上の前提だけを書く(「会員である」)。DB 接続やモック設定などの技術詳細は Given の文言から締め出し、必要なら段の外(setup/builder)へ追い出す。
5. **三段をコメント+空行で対応づける**:`// Given …` `// When …` `// Then …` を業務文で書き、AAA と同じく空行で区切る。コメントがそのまま仕様の読み物になる状態を保つ。
6. **失敗を1回見る**:Then の期待値を崩して走らせ、狙ったシナリオで落ちることを確認してから戻す。

### 完了チェック(もれ確認)
- **Given が業務語か**:Given のコメントと準備が、業務担当者の読める前提になっているか。技術的足場(接続・モック)が Given の文言に紛れていれば追い出す。
- **When が1操作か**:When ブロックの業務操作が1つだけか。2操作あれば Then の原因が曖昧。
- **Then が1結果か**:`grep -n "expect(" <test>` のヒットが1つの業務結果を指しているか。複数の無関係な結果があればシナリオが割れている。
- **シナリオと1対1か**:そのテストが台帳のシナリオ1件に対応し、対応する `T-ID` が書けるか。書けないテストは根拠不明(抽出漏れか過剰縛り)。
- **落として確認したか**:手順6を実行済みか。

### 網羅とのつなぎ方
Given-When-Then は AAA と同型で、網羅対象を受け入れ基準や Gherkin シナリオの側に取る。
つまり網羅すべき集合は、受け入れシナリオの一覧であり、それを一シナリオ一テストへ写す。
シナリオを出すときも、同値分割や境界値で正常系、境界、異常系を出し切ってから書く。
When に操作を二つ以上入れると、どちらの振る舞いを網羅したかが曖昧になり、シナリオとテストの一対一が崩れる。
だから When は一操作に絞り、Then で一つの結果だけを確かめる。
全シナリオに対応するテストが緑になった時点を、受け入れ観点での網羅完了とみなす。

## テーブル駆動 / パラメタライズドテスト(Table-Driven / Parameterized)

### 概要
同じ構造の入力と期待値の組を表(ケース配列)に並べ、1本のテスト定義で全行を回す構成。
行を増やすだけでケースを足せる。

### 目的/いつ使う
同じ操作を入力違いで何度も検証するときに使う。
同値分割の代表値や境界値分析の境界点は、本来この形で実装される([`blackbox-partition.md`](blackbox-partition.md) の `it.each` 例がそのまま当てはまる)。
分岐そのものが入力ごとに違う(値ごとに別経路をたどる)ときは、一表に押し込めず手法を見直す。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";
import { classifyAge } from "./age";

describe("classifyAge: table-driven", () => {
  const cases = [
    { partition: "child", value: 5, expected: "child" },
    { partition: "adult", value: 30, expected: "adult" },
    { partition: "senior", value: 70, expected: "senior" },
    { partition: "invalid", value: -1, expected: "invalid" },
  ] as const;

  it.each(cases)("$partition ($value) -> $expected", ({ value, expected }) => {
    expect(classifyAge(value)).toBe(expected);
  });
});
```

### 落とし穴
- ケース名が静的だと、失敗時にどの行で落ちたか分からない。`$value` や `$partition` で行ごとに動的なラベルを付ける。
- ループ内に `if` で分岐を入れ、行ごとに違う検証をすると、一表が実質複数テストになり一テスト一振る舞いが崩れる。検証が割れるなら表を分ける。

### 遂行手順(着手→完了)
テーブル駆動で書く作業の本体は「同じ検証式で回せる代表値だけを表に集め、各行に逆引きできるラベルを付ける」ことだ。次の順で進める。

1. **同じ操作・同じ検証式で回るか確かめる**:表に入れたいケース群が、入力と期待値だけ違って実行と検証の式は共通か確認する。検証式が行ごとに変わるなら表に向かない(下の手順6で分ける)。
2. **代表値の出所を持ってくる**:行に入れるのは思いつきの値ではなく、同値分割の代表値・境界値分析の境界点([`blackbox-partition.md`](blackbox-partition.md))。抽出済みの値をそのまま行へ写す。
3. **ケース配列を `{入力, 期待値, ラベル}` で組む**:各行に区分名(`partition`)や入力値そのものを持たせ、`as const` で固定する。期待値は行に直書きし、検証式の中で計算しない(計算すると本番ロジックの写経になり乖離する)。
4. **動的ラベルで `it.each` を回す**:`it.each(cases)("$partition ($value) -> $expected", …)` のように、行の値をテスト名に埋め込む。失敗時にどの行かが出力で分かる。
5. **検証式は1つに保つ**:本体は `expect(fn(value)).toBe(expected)` の一式だけ。`if` で行ごとに分岐させない。
6. **検証が割れるケースは別表へ**:正常系と異常系で見るものが違う(値 vs 例外送出)なら、表を2つに分ける。1表に押し込むと一テスト一振る舞いが崩れる。

### 完了チェック(もれ確認)
- **行数と代表値数が一致するか**:手順2で出した同値クラス/境界点の数と、ケース配列の行数が一致するか。少なければ網羅漏れ、多ければ重複。
- **ラベルが動的か**:`grep -n "it.each" <test>` のテンプレ文字列に `$` 付きプレースホルダがあるか。静的名なら失敗行が特定できない。
- **本体に `if` が無いか**:`it.each` のコールバック内に分岐が無いか。あれば行ごとに別検証=表の分割対象。
- **期待値が直書きか**:`expected` が配列に定数で書かれ、検証式内で本番同等の計算をしていないか(計算は写経で乖離源)。
- **異常系が同居していないか**:値検証と例外検証が1表に混在していないか。混在していれば表を分ける。

### 網羅とのつなぎ方
テーブル駆動は、同値分割や境界値で出した代表値の一覧を、そのまま行に写す実装形である。
網羅すべき集合はケース配列の行の集合であり、抽出した代表値の数と行数が一致して初めて網羅が数えられる。
行ごとに動的ラベルを付けておくと、どの代表値が緑かを失敗出力から逆引きでき、網羅の達成度を行単位で追える。
