# テスト戦略: 配分と階層化

テストレベル([`levels.md`](levels.md))で粒度を選んだあと、どのレベルに何本置き、どの軸でテストを階層に割るかを決める戦略をまとめる。
配分の話(ピラミッドとトロフィー)と、粒度とは別軸の階層化(関心事による分割)の2つを扱う。

配分・階層化は「どのテストを何本どこに置くか」の戦略だが、テスト戦略はそれだけでは閉じない。
何を優先して作り(リスク)、どこに差し込み(シフト)、どの環境とデータで動かし(環境・データ)、緑をどう守り続けるか(フレーキー対策・CI ゲート)まで含めて初めて戦略になる。
以降、配分・階層化のあとにこの5領域を足す。

## テストピラミッドとテストトロフィー(配分論)

どのレベルに何本書くかの配分を示す 2 つのモデル。
どちらも「遅く脆いテストを上に積みすぎるな」が共通の主張で、重心の置き方が違う。

テストピラミッド(Mike Cohn)は、土台に速い単体テストを多く、上に行くほど結合や E2E を少なく積む。
実行が速く失敗箇所を切り分けやすいが、モックに頼りすぎると「単体は全部緑なのに繋ぐと壊れる」状態を招きうる。

テストトロフィー(Kent C. Dodds)は、結合テストに最も重みを置く。
「ユーザーの使い方に近いほど確信度が高い」という主張で、実依存をなるべく本物にした結合層を厚くする。
ただし結合を厚くするほど 1 本が遅く不安定になりやすく、土台の静的検査(型、lint)で下支えする前提が要る。

実務上の指針はひとつ。
本数ではなく、各テストが「壊れたら本当に困ること」を検証しているかで配分を決める。
速くて安定なものを多く、遅くて脆いものを少なく置く。
これはどちらのモデルとも矛盾しない。

### 網羅の配分

各レベルの網羅対象は [`levels.md`](levels.md) で定義した。
配分はその対象を踏まえ、本数ではなく「壊れたら困ること」で決める。
ロジック分岐の網羅は速く安定な単体に厚く集約し、接続のずれは結合に代表で、要約的な要件確認は上層に薄く置く。
遅く脆いレベル(システム、E2E)へ網羅を持ち上げると、確信度の割に CI が遅く不安定になる。
重要フローほど上層に最小本数だけ置き、その細部の網羅は下層へ送り返す、を全レベルで徹底する。

## 関心事による階層化(粒度とは別の軸)

テストレベルは**粒度**(対象の広さ: 単体→結合→E2E)で段を分ける軸だった。
それと直交するもう一本の軸が、**関心事**でテストを階層に割る考え方である。
同じ E2E の中にも「URL が正しく描画されるか」「入力検証が受入仕様を満たすか」「変数割り当てが正しいか」という別々の関心が同居しており、これらを一枚のテストに混ぜると、何か一つ変わるたびに無関係な検証まで巻き込んで赤くなる。

### 分割の軸
関心事で割るとき、次の3つを手がかりにする。

- **完結できる単位で切る**：それ単体で合否を言い切れる最小のまとまり(1リクエストの検証処理、1関数の入出力、1コールツリー)を1階層にする。隣の関心を借りないと判定できないなら、まだ混ざっている。
- **境界付けられたコンテキストで切る**：ドメインの文脈境界に沿って分ける。セッションをまたぐ一連操作、単一 URL のレスポンス、変数割り当て、はそれぞれ別コンテキストで、別階層が受け持つ。
- **フローと状態で切る**：「処理が順に流れること(フロー)」と「ある時点の値が正しいこと(状態)」は別の関心。フローの検証で状態の正しさまで背負わせない。状態は状態の階層が、フローはフローの階層が保証する。

### なぜ効くか
- **各階層が独立して機能する**：ある関心を検証する階層は、他階層が保証する事柄を前提にしてよい。例えば「入力値が受入仕様に妥当か」だけを見るテストは、描画も変数生成も他階層に委ね、自分はその一点に集中する。
- **変更の影響を局限できる**：受入仕様が変わってもその関心の階層だけが赤くなり、フローや描画のテストは巻き込まれない。脆さの主因(無関係な変更での赤)が構造的に消える。
- 結果としてフィードバックが速く、並行して触れる。これが「テストを階層化すると開発の機動力が上がる」の中身。

### 落とし穴
- ツールが謳う区分(「これは E2E ツールだからフローも状態も全部見る」)に引きずられ、一枚に関心を詰め込む。ツールの用途でなく、関心事で割る。
- コンテキストが要らない場面まで階層を増やす。階層はプロセスに要求されるコンテキストの有無で増減させる。文脈が一つしか無いなら粒度の軸だけで足りる(YAGNI)。
- 粒度の軸と取り違える。これは「どこまで広く」ではなく「何の関心を」の軸。両軸は掛け合わせて使う(例: 単体レベル × 状態の関心)。

## リスクベーステスト(何から作るか)

抽出した振る舞いを全部いっぺんに同じ厚みでは作れない。
どれから手厚くテストするかを、**欠陥が出る確率 × 出たときの影響**で並べて決めるのがリスクベーステストである。

### 概要・目的

各振る舞いに、欠陥が混入しやすさ(複雑な分岐、頻繁に変わる箇所、新規・未知の技術、過去にバグが出た箇所)と、壊れたときの影響(金額、データ損失、法令、利用者数、復旧コスト)の2軸でスコアを振り、積の大きい順に優先する。
高リスクには手厚い網羅(境界・異常系・mutation・複数レベル)を、低リスクには代表1本を、という濃淡を意図的につける。
「全部を均一にテストする」は時間予算の前で必ず破綻し、結果として高リスクも低リスクも中途半端になる。先に並べて、上から埋める。

これは 0段の抽出台帳(SKILL.md)の優先度欄と直結する。
台帳の各 `T-ID` の優先度は、この確率 × 影響で付ける。優先度を落として「対象外」にするときは、低リスク(低確率かつ低影響)であることが落とす根拠になり、その1行を台帳に残す。

### 落とし穴

- 確率だけ、影響だけで決める。低確率でも影響が壊滅的(資金移動、認証)なら上位に来る。両軸の積で見る。
- リスク評価を一度きりにする。リスクは変わる(新機能投入、過去バグの再発領域)。台帳の優先度は変更のたびに見直す。
- 「低リスクだから書かない」と「異常系だから対象外」を混同する。異常系でも高リスクなら最優先。安易な対象外送りの口実にしない(SKILL.md「やらないこと」)。

## シフトレフト / シフトライト(どこに差し込むか)

テストを工程のどこに置くかの軸。
**左(設計・実装の早い段階)に寄せて欠陥を安く潰す**のがシフトレフト、**右(本番)に寄せてテスト環境では再現しない問題を捉える**のがシフトライト。両方使う。

### 概要・目的

シフトレフトは、要件・設計レビュー、型と lint、TDD、契約テストなど、コードが本番に届く前の検査を厚くする。欠陥は下流に流れるほど直すのが高くつくので、上流で止める。
シフトライトは、テスト環境では再現できない本番固有の条件(実トラフィック、実データ分布、実依存の障害)を本番側の観測で捉える。次の3つを「右にずらしたテスト」として位置づける。

- **本番監視・可観測性**:ログ・メトリクス・トレースで、利用者が踏む前に劣化や異常を検知する。アサーションの置き場所が本番という発想。
- **合成監視(Synthetic Monitoring)**:本番へ定期的に擬似リクエストを投げ、主要フローの生死を継続確認する。E2E を本番で回し続けるのに近い。
- **カナリアリリース**:新版を一部トラフィックにだけ流し、指標が悪化したら全面展開前に止める。本番の一部を使った段階的な受け入れテストとして働く。

カナリア・合成監視の詳細(技法としての記述)は [`process-static.md`](process-static.md) 群B(プロセスと運用)を参照。
ここではそれらを「テストを右にずらす戦略上の選択肢」として位置づける。

### 落とし穴

- シフトライトをシフトレフトの代わりにする。本番で捕まえるのは最後の網であって、上流の検査を省く言い訳にしない。安く直せるものは左で潰す。
- カナリアや合成監視に「異常時に止める・気づく」仕掛けを付けない。閾値と自動ロールバック(または通知)が無ければ「流しただけ」で、非機能で「走らせただけ」を戒めたのと同じ穴に落ちる。
- 監視を入れて満足し、何を異常とみなすか(SLO/しきい値)を決めない。判定基準の無い計測は緑赤を言えない。

## テスト環境戦略(どこで動かすか)

テストをどの環境で走らせるかの戦略。
**本番から遠いほど速く安いが乖離で見逃す**、**本番に近いほど確信度が高いが遅く高くつく**、のトレードオフを階層で配分する。

### 概要・目的

- **ローカル / dev**:開発者の手元と CI で回す単体・結合の主戦場。速く、依存は test double か軽量な実物(後述の Testcontainers)で代替する。
- **staging / 本番相当**:本番に近い構成データ・依存・設定で、E2E や受け入れ、性能の一部を回す。設定差(環境変数、権限、ネットワーク)に起因する「dev では通るのに本番で落ちる」をここで捕まえる。
- **ephemeral 環境**:PR ごとに使い捨ての環境を立て、マージ前にその差分だけを本番相当で検証して壊す。長命な共有 staging が抱える「誰かの変更で汚れて緑が信用できない」問題を、隔離と短命さで回避する。
- **Testcontainers**:DB・メッセージキュー・外部依存を実物のコンテナとして単体・結合テスト内から起動・破棄する。モックより本物に近い確信度を、専用環境を持たずローカルと CI で得る。test double との使い分けは [`test-doubles.md`](test-doubles.md) を参照。

戦略は「全レベルを本番相当で回す」ではない。速く安定な層は dev/Testcontainers に厚く、本番相当は重要フローに薄く、配分論(ピラミッド/トロフィー)と同じ重心の置き方をする。

### 落とし穴

- 共有 staging に全テストを集約し、汚染と取り合いで緑が信用できなくなる。隔離が要るものは ephemeral か Testcontainers で持つ。
- 本番相当を名乗りながらデータや設定が乖離していて、確信度の幻想を与える。何が本番と同じで何が違うかを明示しないと、緑の意味が読めない。
- どこでも本番相当を使い、CI が遅く高くなる。乖離で困らない層はわざわざ重い環境に上げない(YAGNI)。

## テストデータ戦略(何を食わせるか)

テストに与えるデータをどう用意し、どう独立・再現可能に保つかの戦略。
データの作り方がいい加減だと、テスト間が汚染し合い、緑赤がデータの状態に依存して flaky になる。

### 概要・目的

- **フィクスチャ / factory / builder**:テストが必要とする入力を宣言的に組み立てる。とくに [`test-structure.md`](test-structure.md) の Test Data Builder は、テストの主旨に関係する値だけを書き、残りは妥当なデフォルトで埋めることで、Arrange を読みやすく壊れにくくする。データ構造が変わってもビルダー1か所の修正で済む。
- **seeding**:DB を既知の初期状態に投入してからテストを回す。各テスト(またはスイート)が自分の前提を自分で用意し、隣のテストが残した状態に依存しない独立性を担保する。
- **生成データ**:property-based / fuzzing 用にランダム生成する([`modern-generative.md`](modern-generative.md))。固定値では出ない入力分布で欠陥を炙る。
- **マスキング(本番データ利用)**:本番に近いデータ分布が欲しいとき、個人情報・機密を不可逆に匿名化してから使う。リアルさと機密保護の両立が目的で、生データをテスト環境へ流さないことが前提(信頼境界を越えるデータの扱いはセキュリティ要件であり、ここは省略しない)。

要点は**独立性と再現性**。各テストは自分の前提データを自分で用意し、終わったら片付け、何回・どの順で回しても同じ結果になる状態を保つ。

### 落とし穴

- テスト間で可変なグローバルデータを共有し、実行順で緑赤が変わる。各テストが前提を自前で seeding し、後始末する。
- 巨大な共有フィクスチャに全テストが相乗りし、1か所いじると無関係なテストが連鎖崩壊する。builder で各テストに必要な分だけ組む。
- 本番データを匿名化せずテスト環境へ持ち込む。マスキングを挟まない本番データ利用は機密漏洩であり、リアルさのための近道にしない。

## フレーキー対策・隔離方針(緑を信用できる状態に保つ)

不安定に成否が揺れるテスト(flaky)は、緑の信頼そのものを壊す。
「どうせたまに落ちる」が常態化すると、本物の赤も見逃される。
戦略レベルでは、flaky を**検出 → 隔離 → 原因修正**の流れに乗せる方針を持つことだけを決める。

### 概要・目的

flaky を放置も即削除もしない。検出して可視化し、本流の緑を汚さないよう一時隔離し、原因(時刻・乱数・並行・順序依存・外部依存・待ち合わせ不足)を突き止めて直し、本流へ戻す。
隔離はあくまで一時退避であって、隔離したまま放置は「守られていないテスト」を増やすだけ(SKILL.md の skip 黙殺の罠と同じ)。期限を切って原因修正へ送る。

詳細な検出手法・隔離の具体策・根本原因の分類と対処は [`test-flakiness.md`](test-flakiness.md) に委ねる。
ここでは「flaky を戦略の中で検出・隔離・修正のループに乗せる」という方針までに留める。

### 落とし穴

- リトライで握り潰して原因を見ない。再試行は一時しのぎで、根本原因(競合、時刻依存)はリトライでは消えない。
- 隔離したまま戻さず、隔離置き場が永久の墓場になる。期限と担当を付けて修正へ送る。
- flaky を全部 skip して緑を取り繕う。守られないテストが増えるだけで、抽出台帳の右側が見せかけで埋まる。

## CI ゲーティング / 並列化 / 実行時間予算(緑を守り続ける)

緑にしたテストは、継続実行に載せて初めて守られる(SKILL.md ワークフロー4)。
CI でどの基準をマージのゲートにし、どう速く回し、どこまで時間を許すかを戦略として決める。

### 概要・目的

- **CI ゲーティング**:コミット・マージのたびに自動実行し、退行を早期に捕まえる。さらにカバレッジと mutation([`mutation-testing.md`](mutation-testing.md))に**下限**を設け、下限を割る変更をマージさせない。目標値の決め方は「全コードに高い率を一律要求」ではなく、リスクベースで critical な領域(金額・セキュリティ・状態遷移)に厳しい下限、それ以外は緩めに、と濃淡を付ける。カバレッジは到達を、mutation は検出力を見るので、critical には両方を当てる。
- **並列化**:テストを分割して並行実行し、総実行時間を縮める。並列化は共有リソースの直列化や命名衝突を新たに招くので、並列前提のデータ独立性([`test-doubles.md`](test-doubles.md)・テストデータ戦略)が要る。複数 subagent への並列委譲時の罠は [`../../_shared/parallel-delegation.md`](../../_shared/parallel-delegation.md) を参照。
- **実行時間予算**:CI 全体・各レベルに「ここまでの時間」という予算を置く。予算を超えたら、遅いテストを下層へ送り返す(E2E の網羅を単体へ移す)、並列度を上げる、頻度を下げる(全件は nightly、PR は影響範囲だけ)のいずれかで収める。予算が無いと遅いテストが無制限に積もり、CI が遅すぎて誰も待たなくなる(=実質ゲートが死ぬ)。

### 落とし穴

- カバレッジ率自体を目的化する。高い率は到達を示すだけで検証を保証しない(SKILL.md)。下限は最低ラインであって、達成を品質の証明にしない。mutation で検出力を併せて見る。
- 下限を一律に上げて、低リスク領域のために高コストなテストを強制する。ゲートの厳しさはリスクで配分する(YAGNI)。
- 実行時間予算を決めず、遅い E2E を上層に積み増す。CI が遅くなるほど開発者は緑を待たず回避し、ゲートが形骸化する。重要フローは上層に最小本数、細部の網羅は速い下層へ(配分論と同じ)。
