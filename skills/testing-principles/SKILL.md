# テストの7原則(なぜこの技法群が要るかの理由付け)

JSTQB/ISTQB が定めるテストの7原則。個々の技法(同値分割、mutation、リスクベーステスト等)は
このリポジトリの各 reference に散らばっているが、**なぜその技法が要るのか**の理由付けを1箇所にまとめる。
理由が見えないと、忙しいときに「省いていい手順」と誤解されやすい。

各原則は「原則そのもの」「この技法カタログでの対応先」の構成で示す。

## 1. テストは欠陥があることを示せるが、欠陥が無いことは示せない

テストは欠陥の存在を示せるが、欠陥ゼロの証明にはならない。「テストが全部緑=バグが無い」という誤った安心を生む。

- 対応先: [`mutation-testing.md`](../mutation-testing/SKILL.md) が「テストの強さ」を測るのはこの原則の裏返し。緑を通っても、
  意図的に注入した欠陥(ミュータント)を検出できなければ、そのテストは実際の欠陥も見逃す可能性が高い。

## 2. 全数テストは不可能

入力の組合せ・状態・経路は現実的な時間で全て網羅できない。テストは有限のリソースの中でリスクに基づき選択する。

- 対応先: [`strategy-allocation.md`](../strategy-allocation/SKILL.md) のリスクベーステスト(確率×影響でのスコアリング)。
  [`blackbox-partition.md`](../blackbox-partition/SKILL.md) の同値分割・境界値分析は「全数の代わりに代表値で網羅とみなす」技法そのもの。

## 3. 早期テストで時間とコストを節約する

欠陥は開発の後工程で見つかるほど修正コストが跳ね上がる。テスト活動はコーディング前(要件・設計段階)から始める。

- 対応先: `../SKILL.md` の 0段抽出ワークフロー(実装前にテストすべき振る舞いを洗い出す)がこの原則の実践。
  [`process-static.md`](../process-static/SKILL.md) の静的レビュー・静的解析も、実行前に欠陥を検出する早期テストの一形態。

## 4. 欠陥はモジュールに偏在する

欠陥は全モジュールに均等に分布せず、一部のモジュールに集中する傾向がある(複雑度が高い、変更頻度が高い等)。

- 対応先: [`strategy-allocation.md`](../strategy-allocation/SKILL.md) のリスクベーステストが、この偏在を根拠にテスト配分の濃淡を付ける。
  複雑度が高いモジュールほど厚くテストする判断の裏付け。

## 5. 殺虫剤のパラドックス

同じテストを繰り返すだけでは、そのテストで検出できる種類の欠陥は既に潰れており、新しい欠陥を見つけられなくなる。
テストは定期的に見直し、更新する必要がある。

- 対応先: [`mutation-testing.md`](../mutation-testing/SKILL.md) が存在する直接の理由。既存テストが本当に欠陥を検出する力を保っているかを、
  人工的な変異注入で定期的に検証する。[`experience-checklist.md`](../experience-checklist/SKILL.md) のエラー推測・[`experience-exploratory.md`](../experience-exploratory/SKILL.md) の探索的テストも、
  体系的技法だけでは尽くせない新しい欠陥パターンを狙う補完策。

## 6. テストは状況次第

正しいテストの手法・厳密さは、対象のドメイン、リスク、契約、法規制によって変わる。すべての対象に同じ厳密さを適用しない。

- 対応先: `../SKILL.md` の役割分担(通常は TDD、状態遷移や並行があれば loop-engineering、critical なら formal-verification)は、
  この原則の具体化。[`nonfunctional-*.md`](../nonfunctional-attributes/SKILL.md) の非機能テストも対象の性質に応じて選択的に適用する。

## 7. 「バグゼロ」の落とし穴

欠陥をゼロにしても、それだけでは使えないシステム・要求に合わないシステムができることがある。テストは仕様準拠だけでなく、
実際に使う人の期待に応えているかも見る必要がある。

- 対応先: [`levels-system.md`](../levels-system/SKILL.md) の受け入れテスト(UAT)、[`nonfunctional-attributes.md`](../nonfunctional-attributes/SKILL.md) の
  ユーザビリティテストが、この原則の実践(仕様通りだが使えない、を防ぐ層)。

---

これらの原則は個々の reference に技法として実装されているが、**なぜその技法を省いてはいけないか**を判断するときの拠り所として、
迷ったらこのファイルに立ち戻る。
