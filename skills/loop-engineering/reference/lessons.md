# 抽出漏れの教訓(実地で踏んだ)

RFC 6455 WebSocket 実装で、**TLA+ も Lean も TDD も緑なのに、複数の RFC MUST 要件(RSV ビット拒否、最小長符号化拒否、受信 close code 検証)が実装から漏れていた**。
後から監査で発覚。
原因は3つで、いずれも0段(抽出)の欠落だった。

- **TLA+ の抽象化が死角を作った**: WsStream spec は設計を固めるためフレームのバイト中身(RSV、close code の値、長さの符号化形式)を抽象化して捨てていた。これは**正しい設計判断**(状態機械にバイトは不要)だが、捨てた領域は TLA+ の検査範囲に原理的に入らない。**抽象化の境界 = 検証の死角**。
- **Lean は「正しさ」を証明するが「使われている」は証明しない**: `closeCodeSendable` は Lean P7 で証明済みだったが、**受信経路から呼ばれていなかった**(デッドコード)。Lean は呼び出しグラフを見ないので証明は緑のまま。「証明した」と「実装が使う」の間の橋(TDD の結線テスト)が無かった。
- **要件の親リストが無かった**: そもそも RSV/最小長は EARS 要件にも Lean P番号にも**最初から存在しなかった**。close code は Lean には入ったが EARS には入らなかった(片系列のみ=非対称漏れ)。元仕様(RFC)→要件の抽出が「設計検証に必要な分」に偏り、状態に絡まないバイト検証を取りこぼした。

教訓: **層分け(TLA+/Lean/TDD)は冗長を消すが、「各層が何を見ないか」という死角を生む。穴は層の内側でなく、層の境界=抽象化で捨てた領域に落ちる。**
0段のトレーサビリティ・マトリクスがあれば、「RFC§5.2 RSV の行が全列空」「P7 はあるが要件列が空」が一目で出て、3つとも入口で捕まった。
さらに出口に Autobahn を置けば、すり抜けても実挙動で落ちた。
**形式手法は抽出された要件を厳密に検証するが、抽出そのものの網羅は別レイヤー(0段)の責任。**
