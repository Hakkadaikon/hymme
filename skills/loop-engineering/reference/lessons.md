# 抽出漏れの教訓(実地で踏んだ)

**TLA+ も Lean も TDD も緑なのに、元仕様の複数の MUST 要件が実装から漏れていた**ケースが実際にあった。
後から監査で発覚。原因は3つで、いずれも0段(抽出)の欠落だった。ドメインは問わない(プロトコル、API、
状態機械を持つ業務ロジックいずれでも同じ形で起きる)。

- **形式手法の抽象化が死角を作った**: 設計を固めるための spec は、状態遷移に関係ない詳細(値の内部表現、
  エンコーディング形式、個々のフィールドの妥当性)を意図的に捨てている。これは**正しい設計判断**
  (状態機械に不要な詳細は要らない)だが、捨てた領域は形式検査の範囲に原理的に入らない。
  **抽象化の境界 = 検証の死角**。
- **証明は「正しさ」を保証するが「使われている」は保証しない**: ある性質が Lean(または他の証明系)で
  証明済みでも、**その証明対象の関数が実際の呼び出し経路から呼ばれていない**(デッドコード)ことがある。
  証明系は呼び出しグラフを見ないので証明は緑のまま。「証明した」と「実装が使う」の間の橋
  (TDD の結線テスト)が無いと、この非対称は誰にも気づかれない。
- **要件の親リストが無かった**: そもそも漏れた要件が EARS 要件にも証明対象のリストにも**最初から
  存在しなかった**。ある性質は証明系には入ったが EARS には入らなかった(片系列のみ=非対称漏れ)。
  元仕様→要件の抽出が「設計検証に必要な分」に偏り、状態遷移に絡まない検証(値の妥当性チェック等)を
  取りこぼした。

教訓: **層分け(TLA+/Lean/TDD)は冗長を消すが、「各層が何を見ないか」という死角を生む。穴は層の内側でなく、
層の境界=抽象化で捨てた領域に落ちる。**
0段のトレーサビリティ・マトリクスがあれば、「この仕様条項の行が全列空」「この証明はあるが要件列が空」が
一目で出て、3つとも入口で捕まる。さらに出口に既製の適合性テストスイート(あるドメインだけ)や
ファジング・差分テストを置けば、すり抜けても実挙動で落ちる。
**形式手法は抽出された要件を厳密に検証するが、抽出そのものの網羅は別レイヤー(0段)の責任。**

## 実例(参考)

上記は RFC 6455(WebSocket)の実装で実際に踏んだ教訓を一般化したもの。具体的には、フレームの RSV ビット拒否、
最小長符号化拒否、受信 close code 検証という3つの RFC MUST 要件が、それぞれ上記3パターンの原因で
実装から漏れていた(spec のバイト表現の抽象化、証明はあったが受信経路未結線、要件抽出時の取りこぼし)。
