## シフトレフト / シフトライト(どこに差し込むか)

テストを工程のどこに置くかの軸。
**左(設計・実装の早い段階)に寄せて欠陥を安く潰す**のがシフトレフト、**右(本番)に寄せてテスト環境では再現しない問題を捉える**のがシフトライト。両方使う。

### 概要・目的

シフトレフトは、要件・設計レビュー、型と lint、TDD、契約テストなど、コードが本番に届く前の検査を厚くする。欠陥は下流に流れるほど直すのが高くつくので、上流で止める。
シフトライトは、テスト環境では再現できない本番固有の条件(実トラフィック、実データ分布、実依存の障害)を本番側の観測で捉える。次の3つを「右にずらしたテスト」として位置づける。

- **本番監視・可観測性**:ログ・メトリクス・トレースで、利用者が踏む前に劣化や異常を検知する。アサーションの置き場所が本番という発想。
- **合成監視(Synthetic Monitoring)**:本番へ定期的に擬似リクエストを投げ、主要フローの生死を継続確認する。E2E を本番で回し続けるのに近い。
- **カナリアリリース**:新版を一部トラフィックにだけ流し、指標が悪化したら全面展開前に止める。本番の一部を使った段階的な受け入れテストとして働く。

カナリア・合成監視の詳細(技法としての記述)は [`process-static.md`](../process-static/SKILL.md) 群B(プロセスと運用)を参照。
ここではそれらを「テストを右にずらす戦略上の選択肢」として位置づける。

### 遂行手順(着手→完了)

差し込み位置は「各検証を、それを最も安く確実に潰せる工程へ割り付ける」作業だ。次の順で進める。

1. **各検証を「本番前に潰せるか / 本番でしか出ないか」で二分する**:型・lint・契約・ロジックの網羅は本番前(左)に潰せる。実トラフィック・実データ分布・実依存の障害に起因するものは本番(右)でしか出ない。台帳の `T-ID` を見てどちらかへ振る。
2. **左へ振ったものを上流の検査へ割り付ける**:要件・設計レビュー、型/lint、TDD、契約テストのどれで止めるかを決める。下流に流すほど修正コストが上がるので、最も早い工程に置く。
3. **右へ振ったものを観測手段へ割り付ける**:本番監視(劣化検知)、合成監視(主要フローの生死)、カナリア(段階展開での指標比較)のどれで捉えるかを決める。技法の詳細は [`process-static.md`](../process-static/SKILL.md) 群B。
4. **右の各手段に判定基準と自動対応を必ず付ける**:右に置く検証はアサーションの置き場所が本番なので、何を異常とみなすか(SLO/しきい値)と、異常時の自動ロールバックまたは通知を必ずセットにする。これが無いと「流しただけ」になる。
5. **左右の重複と漏れを点検する**:左で安く潰せるものを右に丸投げしていないか、本番でしか出ないものを左だけで済ませて見逃していないか、両方向を確認する。

### 完了チェック(もれ確認)

- **左で潰せるものを右に回していないか**:本番前に止められる検証(型・契約・ロジック)を本番監視任せにしていないか。安く直せるものは左へ。
- **右の各手段に判定基準があるか**:本番監視・合成監視・カナリアそれぞれに SLO/しきい値が決まっているか。基準の無い計測は緑赤を言えない。
- **右の各手段に自動対応があるか**:カナリア・合成監視に、異常時の自動ロールバックか通知が付いているか。無ければ「流しただけ」。
- **シフトライトを言い訳にしていないか**:上流の検査(レビュー・型・TDD)を「本番で捕まえるから」と省いていないか。右は最後の網であって左の代替ではない。

### 落とし穴

- シフトライトをシフトレフトの代わりにする。本番で捕まえるのは最後の網であって、上流の検査を省く言い訳にしない。安く直せるものは左で潰す。
- カナリアや合成監視に「異常時に止める・気づく」仕掛けを付けない。閾値と自動ロールバック(または通知)が無ければ「流しただけ」で、非機能で「走らせただけ」を戒めたのと同じ穴に落ちる。
- 監視を入れて満足し、何を異常とみなすか(SLO/しきい値)を決めない。判定基準の無い計測は緑赤を言えない。

## テスト環境戦略(どこで動かすか)

テストをどの環境で走らせるかの戦略。
**本番から遠いほど速く安いが乖離で見逃す**、**本番に近いほど確信度が高いが遅く高くつく**、のトレードオフを階層で配分する。

### 概要・目的

- **ローカル / dev**:開発者の手元と CI で回す単体・結合の主戦場。速く、依存は test double か軽量な実物(後述の Testcontainers)で代替する。
- **staging / 本番相当**:本番に近い構成データ・依存・設定で、E2E や受け入れ、性能の一部を回す。設定差(環境変数、権限、ネットワーク)に起因する「dev では通るのに本番で落ちる」をここで捕まえる。
- **ephemeral 環境**:PR ごとに使い捨ての環境を立て、マージ前にその差分だけを本番相当で検証して壊す。長命な共有 staging が抱える「誰かの変更で汚れて緑が信用できない」問題を、隔離と短命さで回避する。
- **Testcontainers**:DB・メッセージキュー・外部依存を実物のコンテナとして単体・結合テスト内から起動・破棄する。モックより本物に近い確信度を、専用環境を持たずローカルと CI で得る。test double との使い分けは [`test-doubles.md`](../test-doubles/SKILL.md) を参照。

戦略は「全レベルを本番相当で回す」ではない。速く安定な層は dev/Testcontainers に厚く、本番相当は重要フローに薄く、配分論(ピラミッド/トロフィー)と同じ重心の置き方をする。

### 遂行手順(着手→完了)

環境戦略は「各テスト群を、確信度と速度の釣り合う最も安い環境へ置く」作業だ。次の順で各層に何を置くかを決める。

1. **テスト群を確信度の要求で仕分ける**:乖離(SQL 方言・設定差・実依存の障害)で困らない論理網羅か、本番固有の設定/データ/依存に依存するかで分ける。前者は速い環境へ、後者は本番に近い環境へ寄せる。
2. **dev/ローカルに論理網羅を厚く置く**:同値分割・境界値・条件網羅など、test double や軽量実物で取れる網羅をここへ集約する。CI でもこの層を主戦場にする。
3. **管理下のプロセス外依存は Testcontainers へ置く**:自前 DB・キューなど、フェイクでは契約が乖離するものを実物コンテナで取る(配分・対象の絞りは [`test-doubles.md`](../test-doubles/SKILL.md) の Testcontainers 手順)。専用環境を持たず dev/CI で確信度を上げる。
4. **設定差で出る不具合は staging/本番相当へ置く**:環境変数・権限・ネットワークに起因する「dev では通るのに本番で落ちる」を捕まえる E2E・受け入れ・性能の一部をここへ最小本数で置く。
5. **隔離が要るものは ephemeral へ置く**:共有 staging の汚染・取り合いで緑が信用できなくなるものは、PR ごとの使い捨て環境で差分だけ検証して壊す。
6. **各環境で「何が本番と同じで何が違うか」を1行明記する**:本番相当を名乗る環境のデータ・設定・依存のうち、本番と一致する点と乖離する点を書く。書けないなら確信度の意味が読めない。

### 完了チェック(もれ確認)

- **論理網羅が速い層にあるか**:広い同値分割・境界値が staging や E2E に上がっていないか。dev/Testcontainers に集約されているか。
- **本番相当が重要フローに絞られているか**:全テストを本番相当で回していないか。重い環境に上げているのが設定差で本当に困るものだけか。
- **本番相当の乖離が明示されているか**:本番相当環境ごとに「本番と同じ点/違う点」が1行書かれているか。書かれていなければ緑の意味が読めない(確信度の幻想)。
- **共有環境に全部集約していないか**:隔離が要るテストが共有 staging に相乗りして汚染し合っていないか。隔離対象が ephemeral か Testcontainers に分離されているか。

### 落とし穴

- 共有 staging に全テストを集約し、汚染と取り合いで緑が信用できなくなる。隔離が要るものは ephemeral か Testcontainers で持つ。
- 本番相当を名乗りながらデータや設定が乖離していて、確信度の幻想を与える。何が本番と同じで何が違うかを明示しないと、緑の意味が読めない。
- どこでも本番相当を使い、CI が遅く高くなる。乖離で困らない層はわざわざ重い環境に上げない(YAGNI)。

## テストデータ戦略(何を食わせるか)

テストに与えるデータをどう用意し、どう独立・再現可能に保つかの戦略。
データの作り方がいい加減だと、テスト間が汚染し合い、緑赤がデータの状態に依存して flaky になる。

### 概要・目的

- **フィクスチャ / factory / builder**:テストが必要とする入力を宣言的に組み立てる。とくに [`test-structure-lifecycle.md`](../test-structure-lifecycle/SKILL.md) の Test Data Builder は、テストの主旨に関係する値だけを書き、残りは妥当なデフォルトで埋めることで、Arrange を読みやすく壊れにくくする。データ構造が変わってもビルダー1か所の修正で済む。
- **seeding**:DB を既知の初期状態に投入してからテストを回す。各テスト(またはスイート)が自分の前提を自分で用意し、隣のテストが残した状態に依存しない独立性を担保する。
- **生成データ**:property-based / fuzzing 用にランダム生成する([`generative-property.md`](../generative-property/SKILL.md))。固定値では出ない入力分布で欠陥を炙る。
- **マスキング(本番データ利用)**:本番に近いデータ分布が欲しいとき、個人情報・機密を不可逆に匿名化してから使う。リアルさと機密保護の両立が目的で、生データをテスト環境へ流さないことが前提(信頼境界を越えるデータの扱いはセキュリティ要件であり、ここは省略しない)。

要点は**独立性と再現性**。各テストは自分の前提データを自分で用意し、終わったら片付け、何回・どの順で回しても同じ結果になる状態を保つ。

### 遂行手順(着手→完了)

データ戦略は「各テストが自分の前提を自前で用意し、後始末し、順序非依存にする」状態を作る作業だ。次の順で進める。

1. **必要なデータの作り方を4種から選ぶ**:固定の入力構造ならフィクスチャ/builder、DB の初期状態が要るなら seeding、入力分布で炙りたいなら生成データ([`generative-property.md`](../generative-property/SKILL.md))、本番分布が要るならマスキング済み本番データ。テストの主旨で選ぶ。
2. **builder で「主旨に関係する値だけ」を書く**:Arrange はテストが見たい値だけを明示し、残りは妥当なデフォルトに任せる([`test-structure-lifecycle.md`](../test-structure-lifecycle/SKILL.md) の Test Data Builder)。データ構造が変わってもビルダー1か所で吸収できる形にする。
3. **各テストが前提を自前で用意する**:DB を要するものは seeding でそのテストの初期状態を自分で作る。隣のテストが残した状態を当てにしない。共有の可変グローバルデータに相乗りしない。
4. **後始末を必ず付ける**:テスト後に作ったデータを片付ける(トランザクションのロールバック、スキーマ切り直し等)。片付けが無いとテスト間に状態が漏れる([`flakiness-concurrency.md`](../flakiness-concurrency/SKILL.md) のテスト間順序・状態漏れ)。
5. **本番データはマスキングを挟む**:本番データを使うなら、テスト環境へ流す前に個人情報・機密を不可逆に匿名化する。生データのまま持ち込まない(セキュリティ要件、省略しない)。
6. **順序非依存を実測で確かめる**:テスト順をシャッフルして回し(`vitest --sequence.shuffle` 等)、緑のままかを確認する。落ちれば独立性が崩れている。

### 完了チェック(もれ確認)

- **各テストが前提を自前で用意しているか**:可変なグローバル/共有データに依存しているテストが無いか。隣のテストの残骸を当てにしていないか。
- **後始末が付いているか**:データを作るテストに片付け(ロールバック/スキーマ切り直し)があるか。無ければ状態漏れの種。
- **順序をシャッフルしても緑か**:`--sequence.shuffle` 等で順序を変えて回しても全緑か。落ちれば独立性が崩れている。
- **巨大共有フィクスチャに相乗りしていないか**:多数のテストが1つの巨大フィクスチャに依存し、1か所いじると連鎖崩壊する構造になっていないか。builder で必要分だけ組めているか。
- **本番データにマスキングが掛かっているか**:本番データ利用箇所すべてに不可逆な匿名化が挟まっているか。生データがテスト環境に入っていないか。

### 落とし穴

- テスト間で可変なグローバルデータを共有し、実行順で緑赤が変わる。各テストが前提を自前で seeding し、後始末する。
- 巨大な共有フィクスチャに全テストが相乗りし、1か所いじると無関係なテストが連鎖崩壊する。builder で各テストに必要な分だけ組む。
- 本番データを匿名化せずテスト環境へ持ち込む。マスキングを挟まない本番データ利用は機密漏洩であり、リアルさのための近道にしない。

## フレーキー対策・隔離方針(緑を信用できる状態に保つ)

不安定に成否が揺れるテスト(flaky)は、緑の信頼そのものを壊す。
「どうせたまに落ちる」が常態化すると、本物の赤も見逃される。
戦略レベルでは、flaky を**検出 → 隔離 → 原因修正**の流れに乗せる方針を持つことだけを決める。

### 概要・目的

flaky を放置も即削除もしない。検出して可視化し、本流の緑を汚さないよう一時隔離し、原因(時刻・乱数・並行・順序依存・外部依存・待ち合わせ不足)を突き止めて直し、本流へ戻す。
隔離はあくまで一時退避であって、隔離したまま放置は「守られていないテスト」を増やすだけ(SKILL.md の skip 黙殺の罠と同じ)。期限を切って原因修正へ送る。

詳細な検出手法・隔離の具体策・根本原因の分類と対処は [`flakiness-concurrency.md`](../flakiness-concurrency/SKILL.md) に委ねる。
ここでは「flaky を戦略の中で検出・隔離・修正のループに乗せる」という方針までに留める。

### 遂行手順(着手→完了)

戦略レベルでやるのは「flaky を検出 → 隔離 → 原因修正 → 復帰のループに乗せる仕組みを置く」ことだ。次の順で進める。具体的な原因別の直し方は [`flakiness-concurrency.md`](../flakiness-concurrency/SKILL.md)。

1. **検出の仕組みを置く**:CI で各テストの成否履歴を記録し、同一コミットで揺れるテストを flaky として可視化する。1回の緑/赤では flaky は見えないので、複数回または継続実行の履歴で判定する。
2. **検出したものを一時隔離する**:flaky と判定したテストを本流の緑から外す(quarantine タグ等)。本流を汚さないためであって、消すためではない。
3. **隔離に期限と担当を付ける**:隔離した各テストに「いつまでに・誰が原因修正するか」を必ず付ける。期限の無い隔離は墓場になる。
4. **原因を分類して直す**:時刻・乱数・並行・順序依存・外部依存・待ち合わせ不足のどれかへ分類し、[`flakiness-concurrency.md`](../flakiness-concurrency/SKILL.md) の対処で根本原因を消す。リトライで握り潰さない。
5. **復帰させる**:原因を直したテストを複数回(または CI の連続成功)走らせて安定を確認し、本流へ戻す。隔離置き場に残さない。

### 完了チェック(もれ確認)

- **検出の仕組みがあるか**:flaky を成否履歴で可視化する仕掛けが CI にあるか。人の記憶頼みになっていないか。
- **隔離が一時退避になっているか**:隔離したテストすべてに期限と担当が付いているか。期限切れで放置されているものが無いか(墓場化の検出)。
- **リトライで握り潰していないか**:flaky をリトライ回数の引き上げだけで黙らせ、原因を見ていないテストが無いか。
- **skip で取り繕っていないか**:flaky を全部 skip して緑を見せかけていないか。skip された行は守られていない(抽出台帳の右側が見せかけで埋まる)。
- **復帰の確認が実測か**:本流へ戻したテストが、複数回または連続成功で安定を実測されているか。1回の緑で戻していないか。

### 落とし穴

- リトライで握り潰して原因を見ない。再試行は一時しのぎで、根本原因(競合、時刻依存)はリトライでは消えない。
- 隔離したまま戻さず、隔離置き場が永久の墓場になる。期限と担当を付けて修正へ送る。
- flaky を全部 skip して緑を取り繕う。守られないテストが増えるだけで、抽出台帳の右側が見せかけで埋まる。

## CI ゲーティング / 並列化 / 実行時間予算(緑を守り続ける)

緑にしたテストは、継続実行に載せて初めて守られる(SKILL.md ワークフロー4)。
CI でどの基準をマージのゲートにし、どう速く回し、どこまで時間を許すかを戦略として決める。

### 概要・目的

- **CI ゲーティング**:コミット・マージのたびに自動実行し、退行を早期に捕まえる。さらにカバレッジと mutation([`mutation-testing.md`](../mutation-testing/SKILL.md))に**下限**を設け、下限を割る変更をマージさせない。目標値の決め方は「全コードに高い率を一律要求」ではなく、リスクベースで critical な領域(金額・セキュリティ・状態遷移)に厳しい下限、それ以外は緩めに、と濃淡を付ける。カバレッジは到達を、mutation は検出力を見るので、critical には両方を当てる。
- **並列化**:テストを分割して並行実行し、総実行時間を縮める。並列化は共有リソースの直列化や命名衝突を新たに招くので、並列前提のデータ独立性([`test-doubles.md`](../test-doubles/SKILL.md)・テストデータ戦略)が要る。複数 subagent への並列委譲時の罠は [`../_shared/parallel-delegation.md`](../_shared/parallel-delegation.md) を参照。
- **実行時間予算**:CI 全体・各レベルに「ここまでの時間」という予算を置く。予算を超えたら、遅いテストを下層へ送り返す(E2E の網羅を単体へ移す)、並列度を上げる、頻度を下げる(全件は nightly、PR は影響範囲だけ)のいずれかで収める。予算が無いと遅いテストが無制限に積もり、CI が遅すぎて誰も待たなくなる(=実質ゲートが死ぬ)。

### 遂行手順(着手→完了)

CI ゲートは「マージを止める基準を critical 領域に強く張り、予算内で速く回す」状態を作る作業だ。次の順で進める。

1. **自動実行のトリガを決める**:コミット・PR・マージのどこで何を回すかを決める(PR は影響範囲、マージ前は全件、など)。退行を最も早く捕まえる位置に置く。
2. **ゲート基準にカバレッジと mutation の下限を置く**:カバレッジ([`whitebox-controlflow-basic.md`](../whitebox-controlflow-basic/SKILL.md)・[`whitebox-dataflow-loop.md`](../whitebox-dataflow-loop/SKILL.md))と mutation([`mutation-testing.md`](../mutation-testing/SKILL.md))に下限を設け、下限割れをマージ不可にする。カバレッジは到達、mutation は検出力なので、critical には両方当てる。
3. **下限をリスクで濃淡付けする**:リスクベース([`strategy-allocation.md`](../strategy-allocation/SKILL.md))で critical な領域(金額・セキュリティ・状態遷移)に厳しい下限、低リスク領域に緩い下限を割り当てる。全コード一律の高率にしない。
4. **並列化のためにデータ独立性を担保する**:総実行時間を縮めるためテストを分割並行実行する前に、共有リソースの直列化・命名衝突を潰す(テストデータ戦略・[`test-doubles.md`](../test-doubles/SKILL.md))。subagent 並列委譲の罠は [`../_shared/parallel-delegation.md`](../_shared/parallel-delegation.md)。
5. **実行時間予算を置き、超過時の収め方を決めておく**:CI 全体・各レベルに時間予算を設定し、超過したら「遅いテストを下層へ送り返す / 並列度を上げる / 頻度を下げる(全件 nightly、PR は影響範囲)」のどれで収めるかを先に決める。

### 完了チェック(もれ確認)

- **マージを止めるゲートになっているか**:下限割れが実際にマージを止める設定か。基準はあるが警告だけ、になっていないか。
- **CI ゲートの下限が critical 領域に設定されているか**:金額・セキュリティ・状態遷移の `T-ID` を含む領域に、厳しいカバレッジ下限と mutation 下限の両方が掛かっているか。critical に mutation が無ければ検出力が無保証。
- **下限がリスクで濃淡付いているか**:低リスク領域にまで高率を一律強制していないか。ゲートの厳しさがリスク配分と整合しているか。
- **並列前提の独立性があるか**:並列化しているなら、順序シャッフルや並行実行で落ちないことを実測したか(データ戦略の順序非依存チェックと連動)。
- **実行時間予算があり守られているか**:CI 全体・各レベルに時間予算が定義され、超過時の収め方が決まっているか。遅い E2E が上層に積み増されて予算を食っていないか。

### 落とし穴

- カバレッジ率自体を目的化する。高い率は到達を示すだけで検証を保証しない(SKILL.md)。下限は最低ラインであって、達成を品質の証明にしない。mutation で検出力を併せて見る。
- 下限を一律に上げて、低リスク領域のために高コストなテストを強制する。ゲートの厳しさはリスクで配分する(YAGNI)。
- 実行時間予算を決めず、遅い E2E を上層に積み増す。CI が遅くなるほど開発者は緑を待たず回避し、ゲートが形骸化する。重要フローは上層に最小本数、細部の網羅は速い下層へ(配分論と同じ)。
