# 期待値を過去の出力で代替するテスト

期待値そのものを1つずつ手で書けないとき、過去の出力や別の信頼できる実装に「正しさの判定」を委ねるテスト群。
古典のブラックボックス技法が「この入力でこの出力」を1つずつ固定するのに対し、ここでは正解を直接持たないまま、信頼できる別の基準との突き合わせで合否を決める。

これらは **oracle 問題**(期待値をどう用意するか)への異なる解として読むと筋が通る。
oracle の取り方は大きく3軸に整理できる。

1. **過去の出力を基準に固める**(スナップショット、承認、仕様化)
2. **別の信頼できる実装と突き合わせる**(差分)
3. **入力と出力の関係・抽象モデル・上流の形式手法に委ねる**(メタモルフィック、モデルベース、形式検証連携)

このファイルは 1、すなわち「**過去の自分の出力**」を基準に取る手法を扱う。
2 の別実装基準は [`oracle-differential.md`](oracle-differential.md)、3 の関係・モデル基準は [`oracle-relational.md`](oracle-relational.md) にまとめた。

各手法は **概要 / 目的といつ使うか / TypeScript example / 落とし穴 / 遂行手順(着手→完了)/ 完了チェック(もれ確認)/ 網羅の定義** の構成で示す。

## ゴールデンテスト/スナップショットテスト

### 概要
出力を初回に「ゴールデン(基準)」として保存し、以降は現在の出力との差分で合否を判定する。

### 目的/いつ使う
レンダリング結果、整形出力、コード生成、大きな構造体など、期待値を手で書くより一度確定させて固定したいときに使う。
差分が出たら、バグか意図した変更かを人が判断し、後者なら基準を更新する。
出力が非決定的(時刻、乱数、順序不定)なものは正規化しないと毎回壊れるので、その前処理が無いなら使わない。

### TypeScript example
整形関数の出力をスナップショットに固定する。

```ts
import { describe, it, expect } from "vitest";
import { renderInvoice } from "./invoice";

describe("renderInvoice: snapshot", () => {
  it("matches the golden output", () => {
    const out = renderInvoice({ id: 7, items: [{ name: "pen", qty: 3 }] });
    expect(out).toMatchSnapshot();
  });
});
```

### 落とし穴
- 差分を読まずに `--update` で機械的に承認すると、バグごと基準を上書きする。更新は必ず差分を見てから行う。
- 巨大スナップショットは差分が読めず形骸化する。意味のある単位に分けるか、要点だけを固定する。

### 遂行手順(着手→完了)
この手法の本体は「スナップショットの対象成果物を列挙し、各々に正規化済みのゴールデンを1度だけ確定させ、以降の差分を必ず人が読む運用に載せる」ことだ。次の順で進める。

1. **対象成果物を列挙する**:スナップショットで固定する出力を全部書き出す。レンダリング結果、整形文字列、コード生成物、大きな構造体など。対象ごとに代表入力を同値分割で選ぶ(全入力ではなく、出力形が変わる代表)。列挙を飛ばすと「重要な出力だけ固定し損ねた」漏れが出る。
2. **非決定要素を正規化する**:各出力に含まれる時刻・乱数・UUID・順序不定・絶対パスを、固定値かマスクへ置換する前処理を入れる([`flakiness-value.md`](flakiness-value.md))。**この層が無いまま固定すると毎回赤になり、すぐ無効化される**。正規化を初回承認より先に必ず済ませる。
3. **初回ゴールデンを確定する**:正規化済みの出力を1度だけスナップショットに書き出す。書き出した中身を**人が1度通読**し、その時点で正しいことを確認してからコミットする(初回を見ずに固定すると、最初から間違った基準を正解にしてしまう)。
4. **差分レビュー運用を回す**:以降テストが赤になったら、差分を読んでバグか意図変更かを判断する。バグなら実装を直す。意図変更なら次の更新フローへ進む。
5. **更新フローを通す**:意図変更と判断したときだけ `--update` 等で基準を更新し、**更新後の差分をレビューに含めてコミットする**。差分を見ずに一括更新しない(落とし穴の機械承認)。

### 完了チェック(もれ確認)
- **全対象にゴールデンがあるか**:手順1で列挙した成果物の数と、実在するスナップショット数が一致する。差があれば固定漏れ。
- **正規化が効いているか**:同じ入力で2回連続実行しても差分ゼロか(時刻・乱数を踏むなら2回目で赤になる)。`vitest` を2回回して確かめる。
- **未レビューの自動承認が無いか**:CI やコミットフックで `--update` を無条件実行していないか確認する。**人が差分を見ずに基準が更新される経路があれば、それはバグごと固定する穴**。`grep -rn "update.*snapshot\|--ci=false\|-u\b" <CI設定>` で自動更新の混入を逆引きする。
- **巨大化で形骸化していないか**:1スナップショットが差分を読めない大きさになっていないか。読めないなら意味のある単位へ分割する。

### 網羅の定義
- **網羅基準**:網羅対象(レンダリング、整形、生成の代表入力集合)それぞれにゴールデンが存在し、現出力との差分がゼロ。
- **網羅手順**:
  1. 代表入力を同値分割で選ぶ。
  2. 各出力をスナップショットに固定する。
  3. 差分が出たらバグか意図変更かを人が判断する。
- **達成チェック**:非決定要素(時刻、乱数、順序)を正規化済みか確認する。巨大スナップショットで差分が読めず形骸化していないかを見る。

## 承認テスト(Approval Testing)

### 概要
スナップショットの一般形。
出力(テキスト、画像、任意の成果物)を「承認済み」ファイルとして保持し、現在の出力と突き合わせ、差分を専用のビューアで人がレビューして承認する。

### 目的/いつ使う
レガシーコードの現状を素早く固定して安全網を張りたいとき(characterization test)や、複雑な成果物を都度レビューで通したいときに使う。
判定基準は「人が見て OK と言った状態との一致」なので、仕様が未文書でも始められる。
仕様が明確で期待値を直接書けるなら、承認の往復を挟まずアサーションで書くほうが速い。

### TypeScript example
承認ツール(例: `approvals`)に成果物を渡す。初回は `received` を生成し、人が `approved` にリネーム/承認して基準化する。

```ts
import { describe, it } from "vitest";
import { verify } from "approvals/lib/Providers/Jest/JestApprovals";
import { generateReport } from "./report";

describe("generateReport: approval", () => {
  it("matches approved report", () => {
    // 差分があれば received と approved を並べて提示し、人が承認するまで失敗
    verify(generateReport({ region: "jp", year: 2026 }));
  });
});
```

### 落とし穴
- 承認の主体が曖昧だと「とりあえず承認」が横行し、基準が腐る。誰が何を見て承認したかを運用で担保する。
- 非決定的出力はスナップショット同様、承認前に正規化(日時や ID のマスク)が要る。

### 遂行手順(着手→完了)
承認テストはスナップショットの一般形だが、**対象成果物の網羅判定**と**承認の主体の明示**が肝になる。テキスト・画像・PDF・JSON など任意の成果物を扱えるぶん、「何を承認対象に取るか」を曖昧にすると守ったつもりの穴が残る。次の順で進める。

1. **承認対象の成果物を網羅列挙する**:承認で固定する成果物を、形式(テキスト/画像/構造化データ)と入力条件の組で全部書き出す。レガシー固定(characterization)なら、現状の振る舞い集合を 0 段の台帳([`../SKILL.md`](../SKILL.md) のワークフロー)から引いて、振る舞いごとに1成果物を立てる。**「主要な出力だけ」で済ませず、振る舞いの数だけ承認対象を立てる**のが網羅判定の本体。
2. **非決定要素を正規化する**:各成果物の日時・ID・順序・環境差を承認前にマスクする([`flakiness-value.md`](flakiness-value.md))。画像なら許容差(ピクセル閾値)を決める。正規化が無いと `received` が毎回ぶれて承認が無意味になる。
3. **received を生成する**:初回はテストを走らせて `received`(現在の出力)を生成させる。この時点ではまだ `approved` が無いのでテストは失敗する(これが正しい初回状態)。
4. **人がレビューして approved 化する**:`received` と `approved` を専用ビューアで並べ、**承認の主体(誰が)と根拠(何を見て OK としたか)を記録**してから `approved` にリネーム/承認する。仕様書・チケット番号など根拠を運用ログに残す。「とりあえず承認」を禁じる。
5. **差分レビュー運用と更新フローを回す**:以降 `received` と `approved` に差分が出たらビューアで読み、バグか意図変更かを判断する。意図変更のときだけ、再び主体と根拠を記録して `approved` を更新する。

### 完了チェック(もれ確認)
- **全対象に承認済みスナップショットがあるか**:手順1で列挙した成果物(振る舞いの数)と、実在する `approved` ファイル数が一致する。差があれば承認対象の取りこぼし=網羅判定の穴。台帳の各振る舞いに `approved` が紐づくかを逆引きする。
- **未レビューの自動承認が無いか**:`received` を自動で `approved` にリネームする経路(CI ジョブ、スクリプト、フック)が無いことを確認する。**人のレビューを通らず approved 化される経路があれば、承認の意味が消える**。`grep -rn "received.*approved\|approve.*all\|auto.*approve" <スクリプト/CI>` で逆引きする。
- **承認の主体と根拠が残っているか**:各 `approved` の更新に、誰が何を根拠に承認したかの記録が紐づくか。記録が無ければ「とりあえず承認」で基準が腐っている疑い。
- **正規化が効いているか**:同じ入力で2回連続実行して `received` が一致するか。ぶれるなら非決定要素のマスク漏れ。

### 網羅の定義
- **網羅基準**:網羅対象の各成果物に承認済み版があり、現在の出力と一致する。
- **網羅手順**:
  1. 対象成果物を列挙する。
  2. received を生成する。
  3. 人がレビューし approved 化する。
- **達成チェック**:承認の主体と根拠が運用で担保されているか、非決定要素をマスク済みかを確認する。レガシー固定(characterization)では現状の振る舞い集合を網羅対象に取る。

## 仕様化テスト(Characterization Testing)

### 概要
仕様書もテストも無い既存コード(レガシーコード)の**現在の実際の挙動**をテストとして固定し、
安全網を張ってからリファクタや機能追加に進む手法(Michael Feathers)。
承認テストの入力を「現状の振る舞い」に絞った特別な使い方であり、TDD のようにグリーンフィールド(新規コード)を前提にしない点が違う。

### 目的/いつ使う
既存コードにテストが薄い/無い状態で、リファクタや変更を安全に行いたいとき。
「テストが無い」を理由に変更を諦めるのではなく、まず現状の挙動を固定してから直す。
**現状の挙動が正しい仕様とは限らない**ことに注意する。バグごと固定してしまうと、そのバグは「仕様」として保護される。

### TypeScript example
現状の出力をまず記録し、テストとして固定してからリファクタに入る。

```ts
import { describe, it, expect } from "vitest";
import { legacyCalculateDiscount } from "./legacy-discount";

// 現状の挙動をそのまま固定する(仕様として正しいかは別途レビュー対象)。
// 各ケースの入力は代表的な呼び出しパターンを実データ/過去ログから拾う。
describe("legacyCalculateDiscount: characterization (現状固定)", () => {
  it.each([
    { qty: 1, price: 100, expected: 100 },   // 割引なし
    { qty: 10, price: 100, expected: 900 },  // 10個で1割引(現状の挙動、意図か要確認)
    { qty: 0, price: 100, expected: NaN },   // 現状 NaN を返す(バグの疑いあり、要レビュー)
  ])("qty=$qty price=$price -> $expected", ({ qty, price, expected }) => {
    const result = legacyCalculateDiscount(qty, price);
    if (Number.isNaN(expected)) {
      expect(result).toBeNaN();
    } else {
      expect(result).toBe(expected);
    }
  });
});
```

### 落とし穴
- 現状の挙動をそのまま「正解」として固定すると、バグごと仕様化してしまう。台帳では「現状はこう動く」と「本来正しいか(要レビュー)」を別列に分け、後者を必ず人がレビューする。
- 固定後に安心してリファクタを止めない。仕様化テストは**リファクタの安全網**であって最終目標ではない。固定できたら実際にリファクタへ進む。
- 網羅せずに一部だけ固定すると、固定していない経路でリファクタが壊れる。

### 遂行手順(着手→完了)
1. **対象コードの呼び出しパターンを列挙する**:実際の呼び出しログ、既存の(あれば)ドキュメント、コードの分岐から「今どう呼ばれているか」を洗い出す。0段の台帳(`../../test-extract/SKILL.md` のワークフロー)を使い、レビュー用途の「既存テストを右側に埋める」形で進めると網羅が上がる。
2. **各呼び出しパターンで現状の出力を記録する**:実際に実行し、出力をそのままテストの期待値として書く(手で予測しない、実行結果を写す)。
3. **「現状はこう動く」と「正しいか要確認」を分けて記録する**:台帳やコメントで両者を別列にする。怪しい挙動(NaN、例外の握りつぶし等)は「要レビュー」に必ず立てる。
4. **固定したテストを緑にしてコミットする**:これで安全網ができた。
5. **要レビュー項目を人に確認する**:バグと判明したものは、まずテストを正しい期待値に直してから実装を直す(Red→Green)。仕様通りと判明したものはそのまま残す。
6. **安全網の上でリファクタや機能追加を進める**:固定したテストが赤くなったら意図しない変更、想定通りの変更ならテストを更新する。

### 完了チェック(もれ確認)
- **主要な呼び出しパターンが全て固定されたか**:手順1の列挙数とテストケース数を突き合わせる。
- **「要レビュー」が放置されていないか**:要レビューに立てた項目が人によって確認され、バグなら修正、仕様なら追認されたか。
- **固定後に実際にリファクタへ進んだか**:安全網を張っただけで終わっていないか(手段が目的化していないか)。

### 「網羅」の扱い
- **網羅基準**:対象コードの主要な呼び出しパターン(実運用で観測される入力の代表)が全て固定されている。
- **停止規範**:実運用ログや既知の呼び出し元を尽くした時点で打ち切る。未知の呼び出しパターンが後で見つかったら追加する。
- **体系的技法への昇格**:固定が終わったら、可能な範囲で同値分割・境界値の観点から抜けを補い、通常のテスト(TDD)に合流させる。
