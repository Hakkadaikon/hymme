# 期待値を関係・抽象モデル・形式手法で代替するテスト

期待値を1つずつ手で書けないとき、入力と出力の**関係**、振る舞いの**抽象モデル**、上流の形式手法で確立した**証明済み述語**に正しさの判定を委ねるテスト群。正解を直接持たないまま、それらの基準との突き合わせで合否を決める。

**oracle 問題**(期待値をどう用意するか)の3つの解のうち、このファイルは関係・形式手法の述語を基準に取る手法(メタモルフィック、形式検証連携)を扱う。抽象モデルを oracle に据えるモデルベーステストは分量が大きいので [`oracle-model-based.md`](oracle-model-based.md) に分けた。過去出力を固める/別実装と突き合わせる解(スナップショット、承認、差分)は [`oracle-snapshot.md`](oracle-snapshot.md) を参照。

各手法は **概要 / 目的といつ使うか / TypeScript example / 落とし穴 / 遂行手順(着手→完了)/ 完了チェック(もれ確認)/ 網羅の定義** の構成で示す。

## メタモルフィックテスト(Metamorphic Testing)

### 概要
単一入力の正解が分からなくても、「入力をこう変換したら出力はこう変わる(変わらない)はず」という関係(metamorphic relation)で検証する。

### 目的/いつ使う
oracle が無い対象(検索ランキング、数値計算、機械学習推論、最適化)で有効である。
正解そのものではなく入力変換と出力の関係を縛るので、期待値を一切用意せずに矛盾を炙り出せる。
出力の絶対値を直接アサートできるなら、わざわざ関係に置き換える必要はない。

### TypeScript example
ソートの metamorphic relation: 入力を並べ替えても出力は不変、長さも不変。

```ts
import { describe, it, expect } from "vitest";
import fc from "fast-check";
import { sortAsc } from "./sort";

describe("sortAsc: metamorphic relations", () => {
  it("output is invariant under input permutation and preserves length", () => {
    fc.assert(
      fc.property(fc.array(fc.integer()), (xs) => {
        const shuffled = [...xs].reverse();
        expect(sortAsc(shuffled)).toStrictEqual(sortAsc(xs));
        expect(sortAsc(xs)).toHaveLength(xs.length);
      }),
    );
  });
});
```

### 落とし穴
- 関係が弱いと(長さ不変だけ等)中身が壊れていても通る。複数の関係を重ねて締める。
- 思い込みの関係(本当は成り立たない)を入れると偽陽性を量産する。関係の根拠は仕様に置く。
- 数値の関係(可換、結合、分配)を等価で確かめるとき、`toBe`(`Object.is`)は `-0` と `+0` を別物と判定して flaky に落ちる。たとえば分配法則の左辺が `-0`、右辺が `+0` になると、関係は成り立っているのに失敗する。数値の関係比較には `===` か許容誤差を使う(PBT と同じ罠。[`generative-property.md`](generative-property.md) の落とし穴参照)。

### 遂行手順(着手→完了)
メタモルフィックテストの本体は「**メタモルフィック関係(入力変換と、それに対応する期待される出力関係)を仕様から複数選び**、各関係を1つの property に落とす」ことだ。難所は関係の選び方なので、そこを系統立てる。次の順で進める。

1. **メタモルフィック関係を選ぶ**:対象の仕様から「入力をこう変換 → 出力はこうなる/変わらない」という関係を**型から系統的に**洗い出す。代表的な型:
   - **不変(invariant)**:入力を並べ替え/言い換えても出力不変(例: ソートは順序入替で結果不変、検索は同義語クエリで上位集合不変)。
   - **同変(equivariant)**:入力の変換が出力に対応する変換を起こす(例: 全要素に `+k` した配列の合計は元の合計 `+ n*k`、画像を回転したら検出枠も回転)。
   - **単調(monotonic)**:入力を増やすと出力が単調に増/減(例: クエリに語を足すと検索ヒット数は増えない)。
   - **冪等/可逆**:`f(f(x)) = f(x)`、`decode(encode(x)) = x`。
   各関係に**仕様上の根拠**を1行で添える(根拠の無い思い込み関係は偽陽性源)。
2. **関係ごとにテスト化する**:各関係を独立した property として書く。1関係1 property に保ち、変換を施した入力と元入力を両方 SUT に通して、期待される出力関係を assert する。PBT(`fast-check`)で入力を広く生成する([`generative-property.md`](generative-property.md))。
3. **数値比較の罠を避ける**:出力関係を数値で比べるときは `toBe`(`Object.is`)を避け、`===` か許容誤差を使う(`-0`/`+0`・丸め差で flaky に落ちる、上の落とし穴)。
4. **関係を重ねて締める**:弱い関係(長さ不変だけ)では中身の破れを見逃すので、不変・同変・単調を複数組み合わせて中身まで縛る。1つの関係で通っても、別の関係で破れないかを足す。
5. **広い入力で多数試行する**:seed を変えて複数回回し、反例が出たら shrink で最小化して関係か実装かを切り分ける。

### 完了チェック(もれ確認)
- **関係が網羅的に挙がっているか**:不変・同変・単調・冪等/可逆の各型について「この対象に成り立つ関係はあるか?」を1つずつ問うたか。型を1つでも問わずに済ませると関係漏れ。
- **各関係に仕様根拠があるか**:property ごとに、その関係が成り立つ根拠が仕様にあることを示せるか。根拠なしの関係は偽陽性源として落とすか保留する。
- **関係が弱すぎないか**:1つの関係だけ(例: 長さ不変)で中身が壊れていても緑にならないか。mutation([`mutation-testing.md`](mutation-testing.md))を回し、中身を壊す変異が survive しないことで関係の強さを確かめる。
- **数値比較が安全か**:`grep -n "toBe(" <test>` の各ヒットが、`-0`/`+0`・浮動小数の関係比較に掛かっていないか逆引きする。掛かっていれば `===`/許容誤差へ。
- **複数回緑か**:seed を変えて複数回(数十回または CI 連続成功)走っても落ちないか([`../SKILL.md`](../SKILL.md) の実行ゲート)。

### 網羅の定義
- **網羅基準**:対象に成り立つ metamorphic relation 集合を全て検査し、各 relation が広い入力で破れない。
- **網羅手順**:
  1. 入力変換と出力変化の関係を仕様から複数導く。
  2. 各関係を property 化する。
  3. 広い生成器で多数試行する。
- **達成チェック**:関係が弱すぎ(長さ不変だけ等)て中身の破れを見逃さないか確認する。各関係の根拠が仕様にあるか、複数関係を重ねて締めているかを見る。

## 形式検証連携(TLA+/Lean → 述語テスト)

### 概要
上流の形式手法で確立した不変条件や証明済み述語を、そのまま実装に対する property-based test の property として叩き、設計とコードの間を埋める。

### 目的/いつ使う
設計や数学的性質を形式手法で固めた後、その性質が実装でも成り立つことを継続的に確かめたいときに使う。
このリポジトリの姉妹スキルと接続する:

- [`loop-engineering`](../../loop-engineering/SKILL.md)(TLA+ で状態遷移、並行、プロトコルの設計を網羅検査し、反例を Gherkin の受け入れ仕様へ落とす)。
  TLA+ の安全性不変条件(例: 「同時にロックを保持するのは高々1者」)を、実装の操作列に対する property として再表現する。
  反例から生まれた Gherkin シナリオは、そのまま具体例のテストになる。
- [`formal-verification`](../../formal-verification/SKILL.md)(Lean 4 でアルゴリズムやセキュリティ性質を証明)。
  Lean で証明した定理(例: `decode (encode x) = x`、入力検証の健全性、分類の網羅性)を、PBT の property として実装に対して走らせる。

形式手法は性質が「数学的に正しい」ことを保証するが、その性質が**目の前の実装で**成り立つかは別問題。
PBT はその橋渡し層であり、証明済み述語を実装にぶつけて乖離を検出する。
TLA+/Lean を回していない素のロジックに、わざわざこの連携を持ち込む必要はない(YAGNI)。

### TypeScript example
Lean で証明済みの述語「decode は encode の左逆元」を、実装に対する property としてそのまま叩く。

```ts
import { describe, it, expect } from "vitest";
import fc from "fast-check";
import { encode, decode } from "./codec";

// Lean で証明済み: ∀ x, decode (encode x) = x
// 同じ述語を実装に対する property として実行し、設計と実装の乖離を検出する
describe("codec: bridges the proven invariant", () => {
  it("decode is a left inverse of encode (proven in Lean)", () => {
    fc.assert(
      fc.property(fc.jsonValue(), (x) => {
        expect(decode(encode(x))).toStrictEqual(x);
      }),
    );
  });
});
```

### 落とし穴
- 形式モデルと実装の型や粒度がずれると、同じ名前でも別の性質を検査してしまう。述語の対応を1対1で明記する。
- 証明済みだからとテストを省くと、モデルに無い実装上の前提(I/O、並行、リソース)が抜ける。橋渡しテストは省略しない。

### 遂行手順(着手→完了)
この連携の本体は「上流(Lean/TLA+)で**証明済みの述語**を列挙し、各述語を実装に対する property へ**1対1で写す**」橋渡しだ。形式手法側の成果物が前提になるので、まずそれを確定させてから写す。次の順で進める。

1. **証明済み述語を列挙する**:
   - **Lean 側**([`formal-verification`](../../formal-verification/SKILL.md)):証明した `theorem` を全て書き出す(例: `decode (encode x) = x`、`validate x → safe x`、分類の網羅性)。`sorry` の残る定理は「保証済み」に数えない(未証明を緑の根拠にしない)。
   - **TLA+ 側**([`loop-engineering`](../../loop-engineering/SKILL.md)):TLC が検査した安全性不変条件(例: 「ロック保持者は高々1」)と、反例から起こした Gherkin シナリオを書き出す。
2. **述語を実装の語彙へ1対1で写す**:各述語について、形式モデルの型・粒度と実装の型・粒度の対応表を作る。`∀ x` の `x` が実装のどの入力に当たるか、状態が実装のどのオブジェクトに当たるかを明記する。**対応が曖昧なまま名前だけ同じにすると別性質を検査する**ので、ここを1行ずつ詰める。
3. **property として叩く**:
   - Lean の `∀` 定理 → PBT の `fc.property`(入力を広く生成して述語を assert)([`generative-property.md`](generative-property.md))。
   - TLA+ の状態不変条件 → モデルベーステスト([`oracle-model-based.md`](oracle-model-based.md))の `run` 内 invariant として各操作後に照合。
   - Gherkin シナリオ → 具体例テスト(その入力列での1ケース)。
4. **実装固有の前提を別途テストする**:形式モデルに無い I/O・並行・リソース・例外を、橋渡しとは別のテストで埋める(証明済みでも省かない、落とし穴)。
5. **上流と同期させる**:Lean/TLA+ の述語が増減したら、橋渡しテストも増減させる。証明側とテスト側を1対1に保ち続ける。

### 完了チェック(もれ確認)
- **全証明済み述語が写ったか**:手順1で列挙した Lean の `theorem` と TLA+ の不変条件の数と、実装側 property の数が1対1で対応するか。漏れた述語は橋渡しされていない=証明が実装に効いていない。
- **`sorry` を緑の根拠にしていないか**:Lean 側に `sorry` の残る定理を「保証済み」として property 化していないか(`grep -rn "sorry" tasks/fv/` 等で逆引き)。未証明は橋渡し対象外。
- **型・粒度の対応が1対1か**:各 property について、形式モデルの記号と実装の値・状態の対応表があるか。曖昧なら別性質を検査している疑い。
- **実装固有の前提が別テストで埋まっているか**:I/O・並行・リソースなど形式モデルに無い前提のテストが存在するか。
- **複数回緑か**:property/モデルベースは seed を変えて複数回回しても落ちないか([`../SKILL.md`](../SKILL.md) の実行ゲート)。

### 網羅の定義
- **網羅基準**:上流(TLA+/Lean)で確立した不変条件と証明済み述語を全て実装への property として叩き、乖離ゼロ。
- **網羅手順**:
  1. TLA+ の安全性不変条件と Lean の定理を列挙する。
  2. 各述語を実装の操作列や入力に対する property へ1対1で写す。
  3. 反例由来の Gherkin を具体例テストにする。
- **達成チェック**:モデルと実装で型や粒度がずれて別性質を検査していないか確認する。証明済みでも橋渡しテストを省かない。
