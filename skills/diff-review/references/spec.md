# レンズ: 仕様一致 (spec)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・実害ベースの貫通原則・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## severity 判定(貫通原則の読み替え)

本体の貫通原則(実害が diff 内で具体化しているか)を「**依頼と差分の乖離が diff 内で確認できるか**」と読み替えて適用する:

- **must**: 要求された振る舞いが diff のどこにも実装されていない(実装漏れ)/ 要求と異なる挙動になる解釈違いが diff 内で確定しているもの
- **want+ / want**: スコープ外変更の混入、暗黙の受け入れ条件の取りこぼしのうち影響が限定的なもの
- **[PLAUSIBLE]**: 依頼文が曖昧で、乖離か意図的な判断かを diff から確定できないもの(severity と併記)

判定の基準は常に agent prompt に注入された「ユーザーの依頼」。依頼に書かれていない一般的な品質は他レンズの領分で、ここでは扱わない。

## 実装漏れ

依頼を要求項目に分解し、各項目が diff のどこで実現されているかを突き合わせる:

- 明示された要求のうち diff に対応する変更が無い: **must**。どの要求がどこに無いかを明示する
- 要求の一部だけ実装されている(複数条件のうち 1 つだけ、対象ファイルの一部だけ): **must**
- 「〜も」「あわせて」で追加された副次要求の欠落も同列に扱う(依頼文の後半は漏れやすい)

## スコープ外変更

- 依頼に無いファイル・ロジックへの変更が混入している: 混入した変更が挙動を変えるなら **must**、変えないなら **want+**(レビュー・revert の単位を壊す)
- 依頼と無関係なリファクタ・整形・rename の抱き合わせ: **want**
- 依頼の遂行に必然的に伴う変更(import 追加・型の追随修正等)には発火させない

## 要件の解釈違い

- 依頼の文言と diff の挙動が食い違う(対象・方向・条件の取り違え。「A のとき B」を「B のとき A」に等): **must**
- 複数の解釈が成り立つ依頼を一方に倒して実装しており、選んだ解釈が diff からもコミットメッセージからも読み取れない: **want+** に `[PLAUSIBLE]` を併記

## 暗黙の受け入れ条件

明文化されていなくても依頼が当然に含意する条件の取りこぼし:

- 既存の呼び出し元・利用者を壊さない(後方互換の暗黙要求)。壊れる経路が diff 内で成立していれば **must**
- エラー時挙動: 正常系だけ実装され、依頼対象の操作が失敗したときの扱いが未定義: **want+**
- 既存の同種機能と挙動・規約を揃える: **want**
