# レンズ: 初見 (fresh-eyes)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・実害ベースの貫通原則・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## severity 判定(貫通原則の読み替え)

本体の貫通原則(実害が diff 内で具体化しているか)を「**初見のレビュー者が PR 単体(差分 + コミットメッセージ + PR 説明)で正しく判断できない構図が成立しているか**」と読み替えて適用する:

- **must**: コミットメッセージ・PR 説明と差分の実体が矛盾している(誤った理解のままマージされる)
- **want+ / want**: 変更理由が読み取れない、説明の不足、誤読を誘う命名・構成
- **[PLAUSIBLE]**: 文脈があれば自明かもしれないが PR 単体では確認できないもの(severity と併記)

このレンズは意図的に前提知識を捨てる: チャット履歴・口頭の合意・過去のやり取りを知らないレビュー者として PR だけを読む。

## 変更理由の可読性

- なぜこの変更が必要かが差分・メッセージ・説明のどこからも読み取れない: **want+**。特に挙動変更・定数変更・条件の反転は、理由が無いと初見は妥当性を判断できない
- 「とりあえず」「一時対応」等、理由未満の説明しか無い恒久的変更: **want**

## コミットメッセージ・説明との整合

- メッセージ・説明が述べる変更内容と diff の実体が食い違う(fix と書いて機能追加、対象の誤記、説明に無い変更の混入): **must**
- メッセージが差分の一部しか説明していない(残りが黙って同居): **want+**

## 説明の自己完結性

- 「前回の議論の通り」「口頭で合意した方式で」等、PR 外の文脈に依存する説明: **want+**。PR 内で完結する要約への差し替えを求める
- 参照している issue・チケットが判断に必須なのにリンクも要約も無い: **want**

## 誤読を誘う命名・構成

- 名前と実体の乖離(`validate` が保存もする、`is〜` が真偽値以外を返す等)で初見が挙動を取り違える: 取り違えが実害に繋がる構図なら **want+**、紛らわしいだけなら **want**
- 無関係な変更の同居で本質の変更が埋もれ、差分の構成が理解を妨げる: **want**

## 検証手順の欠落

- 挙動変更を伴う PR で、レビュー者が動作確認する手段(実行コマンド・確認手順・期待結果)が PR から分からない: **want**
- 説明に書かれた検証手順が diff の変更後には成立しない(コマンド名・前提の変更で手順が壊れた): **must**
