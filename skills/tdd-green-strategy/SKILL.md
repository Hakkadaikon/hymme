# Green に至る戦略(三角測量・仮実装・明白な実装)

TDD サイクルの Green フェーズで、赤を通すコードをどう書くかの戦略をまとめる。
方針が見えないときの三角測量、確信度に応じた仮実装と明白な実装の使い分けを扱う。
出典は t-wada(和田卓人)の TDD 観点。

サイクル全体の運用(Red→Green→Refactor の回し方、テストリスト先行)は [`tdd-cycle.md`](../tdd-cycle/SKILL.md) に分けた。

各項目は「概要」「目的/いつ使う」「TypeScript example(vitest 想定)」「落とし穴」の構成で示す。

## 三角測量(Triangulation)

### 概要
一般化の方針が見えないとき、二つ目、三つ目の具体例(テスト)を足して、仮実装を本実装へ追い込む技法。

### 目的/いつ使う
最初のテストをハードコードで通したあと、一般解への道筋が自明でないときに使う。
方針が自明なら、わざわざ例を増やさず「明白な実装」で一気に書いてよい。
三角測量はあくまで方針が見えないときの補助輪である。

### TypeScript example
```ts
import { describe, it, expect } from "vitest";
import { add } from "./add";

describe("add: triangulation", () => {
  // 1本目だけなら `return 3` でも通る(仮実装)
  it("1 + 2 = 3", () => expect(add(1, 2)).toBe(3));
  // 2本目を足すと、定数では通らず一般化を強制される
  it("3 + 4 = 7", () => expect(add(3, 4)).toBe(7));
});
```

### 落とし穴
- 方針が自明な箇所でも機械的に二本目を足し、冗長なテストを量産する。
- 二本目が一本目と同じ同値クラスにあり、一般化を強制できていない。異なる代表値を選ぶ。

### 遂行手順(着手→完了)
三角測量の本体は「定数返しでは通らない二本目以降の例を、未踏の同値クラスから選んで足し、実装を一般化へ押し込む」ことだ。

1. **一本目を仮実装で緑にする**:最初の Red を、定数返しなどベタ書きで通す([仮実装](#仮実装と明白な実装fake-it--obvious-implementation))。この時点では一般化しない。
2. **一般化の方針が見えるか判断する**:見えているなら三角測量は不要。次の Red を待たず[明白な実装](#仮実装と明白な実装fake-it--obvious-implementation)で本実装に書き直してよい(機械的な二本目は冗長)。見えないときだけ手順3へ進む。
3. **異なる同値クラスから二本目を選ぶ**:一本目と**別の代表値**(別の同値クラス、または境界の端点)で二本目の Red を書く。同じクラスから取ると定数返しのまま通り、一般化を強制できない(下の完了チェックで弾く)。
4. **二本目を通すために実装を一般化する**:定数では二本とも通らないので、共通する規則を抽出して実装を一般化する。これが仮実装から本実装への昇格になる。
5. **必要なら三本目で確証する**:二本でもまだ規則が一意に定まらない(複数の一般化が両方を通す)なら、三本目の代表値を足して規則を一つに絞り込む。定まったら止める。
   - *完了条件*:定数返しが残っておらず、足した各例が別々の同値クラスを代表し、実装が一般化されている。これ以上例を足しても新しい規則が出ない状態。

### 完了チェック(もれ確認)
- **定数返しが消えたか**:実装にハードコードした戻り値(`return 3` 等)が残っていないか。残っていれば一般化未完で、三角測量の目的を果たしていない。
- **各例が別の同値クラスか**:足したテストの入力が互いに別の同値クラス/端点を代表しているか。同じクラスの重複は一般化を強制せず、網羅も進めない冗長テスト。
- **二本目を消すと赤になるか**:一般化後に二本目を一時的に外して、一本目だけで定数返しに戻しても通ってしまわないか(思考実験でよい)。戻せてしまうなら二本目が効いていない。
- **過剰な三本目を足していないか**:規則がすでに一意に定まっているのに惰性で例を増やしていないか。方針が自明な箇所では明白な実装で済ませる(YAGNI)。

### 網羅とのつなぎ方
網羅の観点では、三角測量は異なる同値クラスの代表で複数例を足し、一般化を強制する操作である。
二本目を一本目と同じ同値クラスから取ると、定数返しのまま通り、網羅は一歩も進まない。
だから二本目以降は、未踏の同値クラスや境界の端点から選ぶ。
こうして足した例の一つひとつが、リスト上の別項目を消化する。
一般化が完成すれば、その同値クラス全体を一例で代表できたことになり、網羅効率が上がる。
方針が自明な箇所では、例を増やさず明白な実装で書けばよく、機械的な二本目は冗長な網羅になる(YAGNI)。

## 仮実装と明白な実装(Fake it / Obvious implementation)

### 概要
Green に至る二つの戦略。
**仮実装**は定数を返すなどベタ書きでまず緑にする。
**明白な実装**は正解が自明なときに本実装を直接書く。

### 目的/いつ使う
不安が大きい、または一般化の形が見えないときは仮実装から入り、三角測量で一般化する。
正解に確信があり書き間違える恐れが小さいときは、明白な実装で一気に書く。
二つは対立せず、確信度に応じて選ぶ。

### TypeScript example
```ts
// 仮実装: まず定数で緑にし、後で一般化する
export const square = (n: number): number => 4;

// 明白な実装: 正解が自明なので直接書く
export const squareObvious = (n: number): number => n * n;
```

### 落とし穴
- 確信もないのに明白な実装へ飛び、緑だが間違ったコードを書く。不安なら仮実装に戻る。
- 仮実装のまま三角測量を忘れ、定数返しが残る。一般化まで進めて初めて完了する。

### 遂行手順(着手→完了)
この戦略の本体は「いま書く Green を、自分の確信度で仮実装と明白な実装に振り分け、仮実装を選んだら必ず一般化まで連れていく」ことだ。

1. **確信度を自己評価する**:いまの Red に対し、一般化された正解が頭に浮かんでいて書き間違える恐れが小さいか。小さいなら明白な実装、大きい/形が見えないなら仮実装。判断に迷ったら安全側(仮実装)に倒す。
2a. **明白な実装を選んだ場合**:本実装を直接書き、テストを走らせて緑を確認する。`T-ID` はこの一歩で一般化込みで埋まる。完了。
2b. **仮実装を選んだ場合**:定数返しなどで一本目を緑にする。**この時点では未完**(下の完了チェックで定数返しの残存を弾く)。
3. **仮実装は三角測量へ繋ぐ**:[三角測量](#三角測量triangulation)の手順で異なる同値クラスの二本目以降を足し、定数返しを一般化へ押し込む。
4. **一般化を確認して閉じる**:定数返しが消え、足した全例が緑になったらサイクル完了として項目を閉じる。
   - *完了条件*:明白な実装なら本実装が緑。仮実装なら三角測量を経て一般化が済み、ハードコードが残っていない。

### 完了チェック(もれ確認)
- **仮実装の置き土産が残っていないか**:仮実装で始めた項目すべてについて、定数返し/ハードコードが一般化に置き換わったか。`T-ID` がチェック済みなのに実装に定数返しが残っていれば、見かけだけの緑。
- **確信なき明白な実装になっていないか**:明白な実装で一気に書いた箇所が、本当に確信のあるものだったか。緑だが境界や異常系の代表で確かめていない一般化は、確信の錯覚を疑い、別代表のテストを一本足して裏を取る。
- **戦略の選択がサイクルを跨いで一貫しているか**:不安な箇所を惰性で明白な実装に流していないか。確信度に応じて都度選び直す(常に同じ戦略に固定しない)。

### 網羅とのつなぎ方
仮実装か明白な実装かの選択は、網羅とは直交する Green の戦略であり、確信度で選ぶ。
網羅範囲を決めるのはリストであって、どちらの戦略で緑にしたかではない。
ただし仮実装のまま三角測量を忘れると、リスト項目を網羅したつもりで定数返しが残る危険がある。
このとき緑のチェックは付くが、実装は一般化されておらず、網羅は見かけだけになる。
網羅が成立するのは、各リスト項目が一般化された実装の上で緑になったときである。
明白な実装で一気に書く箇所は、その一歩で対応するリスト項目が一般化込みで埋まる。
