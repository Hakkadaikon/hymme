# 並列 subagent 委譲の落とし穴(3スキル共通)

formal-verification(証明探索)・loop-engineering(モデル検査/mutation)・test-design
(テスト実装)はいずれも重い作業を複数 subagent に並列で投げることを勧める。
並列化で実際に踏んだ落とし穴と、最初から守る規範をここに集約する。
各スキルからはこの1枚を参照する(規範を3箇所に複製しない)。

原則は1つ。**並列化してよいのは「ファイル競合のない独立作業」だけ。共有リソースを
触る工程は1本に直列化する。** 共有リソースとは git の index・結線/集約ファイル
(テストランナー・ビルド定義・性質カタログ・トレーサビリティ表)・単一バイナリの
グローバル名前空間など、リポジトリ全体で1つしかないものを指す。

## 1. 共有リソースは直列化する

- **git の index はリポジトリ全体で共有される単一リソース**。並列 subagent が
  それぞれ `git add`/`git commit` すると、別ファイルを触っていても片方の add が
  他方の commit に混入する(`git commit -a` 的な全ステージ拾いで顕在化)。
  → 並列 subagent には「**コミットせず**変更を worktree に残すだけ」と指示し、
  結線・コミットは後続の専用 subagent が1本でまとめて行う。どうしても並列で
  コミットするなら各自 `git worktree` で別ツリーを使う(重いので通常は不要)。
- **結線・集約ファイル(ランナー/ビルド定義/カタログ/トレーサビリティ表)を触る
  工程も1本に絞る**。実装は最大並列、結線は1 subagent、最後に台帳更新、の
  3工程パイプラインに固定する。並列と直列をファイル競合の有無で切り分ける。

## 2. 命名衝突は命名時点で grep して予防する

単一バイナリ/単一名前空間にまとまる成果物では、責務を MECE に分けても公開
シンボル名(関数・型・定数・テスト名・spec の定義名)はグローバル一意でなければ
ならない。**MECE は責務の分割であって名前空間の分割ではない。** 衝突は統合(結線)
まで顕在化せず、そこで初めて再定義エラーや別シグネチャ衝突として噴出する。

→ 並列 subagent への指示に「公開名はドメイン名を prefix に含める」「命名前に
既存成果へ同名がないか grep してから命名する」を**最初から**入れる。再発を待たない。

## 3. 単体の緑 ≠ 統合後の緑(自己申告を信用しない)

並列 subagent が各自の作業領域($TMPDIR 等)で「全部通った」と報告しても、
全体に統合すると落ちることがある(ヘルパ/定義/マクロ衝突、公開 API 名衝突、
別ファイルの既存違反の露出)。**部品単体の緑は統合後の緑を保証しない。**

→ 完了判定は **統合後の全体ゲート**で行う。部分ゲート(変更ファイル限定の検査・
結線前の古い緑)は「ビルド/検査対象に入っていないコードを緑」と誤判定する。
- 検証とコミットを**同じシェル行に繋がない**。まず検証だけ実行し exit 0 を目視
  確認してから、別コマンドでコミットする。繋ぐなら `A && B` で短絡させ、`;` で
  繋がない。`検証 | tail && commit` はパイプ全体の exit が tail の成功で 0 になり、
  検証が赤でもコミットが走る — ゲートと出力整形を混ぜない。
- 「数の一致」をゲートに使うなら、**数え方自体が衝突で壊れないことを先に保証**する
  (basename 一意などの暗黙前提を置かない)。

## 4. 非同期 subagent は成果物の実在で進捗を判定する

`run_in_background` の完了通知は失われうるし、同一 task-id の stale な重複通知が
来ることもある。**通知を「進捗」と読まない。** 成果物(生成ファイルのサイズ/
タイムスタンプ・git log・期待ディレクトリの存在)を真とし、通知は補助とする。

- 長時間ファイルを生成しない subagent は stall を疑い、能動的に成果物の有無を
  確認して、止まっていれば kill して再投下する。「通知が来ない=作業中」と解釈しない。
- 数千秒・tool 呼び出し三桁・同一 task-id の重複通知は、子 subagent 側の暴走の兆候。
- 長い検証フローは中間成果(srv.log / probe.log 等)をファイルに落として停止耐性を
  持たせ、最終コミットは独立した1ステップに切り出す。途中で止まっても次の担当が
  事実から再開でき、成果物が一貫した状態を保つ。

## 5. 外部ツール依存の検証は計画段階で可否を確認する

「外部ツール/実機で動かして確かめる」を検証計画に入れる前に、そのツールが
sandbox で実行可能か・必要機能を備えるか・必要権限があるかを**先に**確認する。
塞がれてから慌てない。使えないなら最初から in-process / ループバック往復で実証する
fallback を、後付けでなく計画の第一候補として設計に組み込む。塞がれたら誇張せず
in-process 実証へ切り替える。
