# コード・テスト・コミット・コメントの役割分担(全スキル共通)

test-design・loop-engineering・formal-verification のいずれでも、成果物を書くときに
「どこに何を書くか」が混線すると、同じ情報が重複したり、読み手が探す場所を間違えたりする。
4箇所それぞれの役割を1つに絞る。

| 場所 | 書くこと | 書かないこと |
| --- | --- | --- |
| **コード** | How(どう実現するか、ロジックそのもの) | コードを読めば分かる説明の逐語訳 |
| **テストコード** | What(何を保証するか、振る舞い) | 実装の内部手順のなぞり(実装詳細への結合、[`test-design-quality.md`](../test-catalog/references/test-design-quality.md) の脆さ回避) |
| **コミットログ** | Why(なぜこの変更が要るか、動機) | 差分の逐語訳(「〜を追加」だけで終わらせない) |
| **コードコメント** | Why not(なぜ別の書き方をしなかったか、却下した選択肢・回避した罠) | What/How の説明(コードと重複する) |

## 各スキルでの具体化

- **test-design**: 0段の台帳(`T-ID`)が「何を保証するか」の一覧そのもの。テストコードの `it`/`describe` 名は
  この `T-ID` の振る舞いを説明し、実装の内部手順(呼び出し順序、変数名)をなぞらない。
  [`good-test-principles.md`](../test-catalog/references/good-test-principles.md) の「観察可能な振る舞い」もこの原則の実装面での具体化。
- **loop-engineering**: TLA+ の `Inv`/disjunct 名や EARS の管理番号(`S-001` 等)を本体コード・テスト名・
  コミットに漏らさない(既存規範)のは、これらが「Why not のための内部識別子」であって、コードが
  表現すべき How やテストが表現すべき What ではないため。証明の都合で等価変形したら、その理由
  (Why not: なぜ元の形のまま証明できなかったか)をコメントに1行残す。
- **formal-verification**: 証明対象の定義を変形した理由(Why not)は仕様/コメントに残すが、
  「Lean で証明した」という手法の存在(How の手段)はコミットやドキュメントに書かない
  ([`formal-verification/SKILL.md`](../formal-verification/SKILL.md) の裏方規範)。証明そのものは
  How の説明ではなく Why not(なぜこの実装で安全と言えるか)の裏付けとして機能する。

## 判定に迷ったとき

- コメントを書く前に「これはコードを読めば分かるか?」と自問する。分かるなら書かない(What/How の重複)。
  分からない理由(隠れた制約、過去の罠の回避、あえて選ばなかった別解)があるときだけ書く(Why not)。
- コミットメッセージを書く前に「差分そのものの説明になっていないか」を確認する。「何を変えたか」は
  `git diff` で分かるので、コミットログには「なぜ変えたか」だけを書く。
- テスト名を書く前に「実装の内部構造(関数名、変数名、呼び出し順)が漏れていないか」を確認する。
  漏れていたら、テストが検証する振る舞い(What)の言葉に置き換える。
